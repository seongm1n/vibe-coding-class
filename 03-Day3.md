## 핵심 개념

### 프론트엔드 / 백엔드 / DB 한 줄 설명

- **프론트엔드** = 사용자가 보는 화면 (지금까지 Lovable로 만든 것)
- **백엔드** = 화면 뒤에서 데이터를 처리하는 곳 (오늘 Supabase가 그 역할)
- **데이터베이스** = 데이터가 실제로 쌓이는 창고
- 비유: 식당 = 손님 보는 홀(프론트) + 주방(백엔드) + 냉장고(DB)

### BaaS = 백엔드를 빌려쓰기

- 옛날: 서버 직접 깔고 DB 직접 운영 → 비전공자에겐 불가능
- 지금: Supabase 같은 BaaS가 회원가입·DB·파일 저장을 다 해줌
- "비전공자가 4일 만에 앱 만드는 게 가능해진 결정적 이유"

### 환경 변수 (env)

- API 키 같은 비밀값을 **코드에 직접 적지 않고** 따로 관리
- 노출되면 누군가 내 계정으로 요금 쓸 수 있음 → "비밀번호 같은 것"
- Lovable / Vercel 둘 다 환경 변수 관리 화면이 있음 → 거기에 넣는다

### 배포 = 인터넷에 띄우기

- 지금까진 Lovable 안에서만 동작 → URL 있어도 비공개
- Vercel에 올리면 진짜 도메인이 생기고 누구나 접속 가능
- "내 노트북 꺼도 살아있는 앱"이 된다

## 블록별 실습 내용

### ① 백엔드 개념 + Supabase 프로젝트 생성 (45분)

- 화이트보드/슬라이드로 개념 그림 (10분)
- Supabase 가입 → New Project → 리전은 Seoul 선택 (15분)
- 프로젝트 안에서 확인할 것:
  - Authentication 메뉴
  - Table Editor
  - API Keys (이게 환경 변수에 들어갈 값)
- "여기 있는 키 절대 SNS에 올리지 말 것" 강조

### ② 회원가입 / 로그인 (60분)

- 강사 시연 (20분):
  - Lovable에 "Supabase로 회원가입/로그인 화면 추가해줘. 이메일/비밀번호 방식. URL과 키는 환경변수로." 같은 프롬프트
  - 환경 변수 어디에 넣는지 (Lovable 설정 화면)
  - Supabase Authentication에서 가입한 사용자 확인
- 수강생 실습 (40분): 본인 앱에 동일 적용
- 흔한 실패: 환경 변수 키 오타, Supabase URL 맨 끝 슬래시(`/`) 붙임

### ③ 데이터 저장·불러오기 (50분)

- 어제 정리한 "저장될 데이터 1종" 사용
- 예시 (운동 기록):
  - Supabase Table Editor에서 `workouts` 테이블 만들기 (열: id, user_id, date, exercise, reps)
  - Lovable에 "로그인한 사용자가 운동 기록을 추가하고 자기 기록만 볼 수 있게 해줘" 프롬프트
- 강사 시연 후 수강생 본인 앱에 적용
- **RLS(Row Level Security)** 한 줄 언급: "내 데이터를 남이 못 보게 하는 설정" → 강사가 한 번 켜는 시연

### ④ Vercel 배포 (60분)

- Lovable → GitHub 연동 → Vercel Import
- 환경 변수 Vercel에도 동일하게 등록
- 첫 배포 → URL 확인 → 핸드폰으로 접속 시연
- "축하합니다, 인터넷에 본인 앱이 떴습니다" 순간 → 사진 찍기 권장
- 트러블슈팅: 배포 실패 시 로그 어디 보는지, 환경 변수 빠뜨린 경우 대응

## 시연 코드 / 예제

- Supabase 테이블 스키마 예시 1~2개 슬라이드로 준비
- "이 프롬프트 그대로 써도 됨" 템플릿 출력본 배포
  - 회원가입 추가 프롬프트
  - CRUD 추가 프롬프트
  - Vercel 배포용 점검 체크리스트

## 강사용 진행 메모

- **오늘이 가장 막히는 날**: 환경 변수, OAuth, 배포 에러가 한 번에 쏟아짐 → 진행 속도 절반으로 떨어진다고 가정
- **회사 동료 보조 인력 활용**: 강사는 전체 진행·시연, 동료는 막힌 사람 1:1 트러블슈팅 분산 담당. 사전에 환경 변수·OAuth·배포 에러 흔한 케이스 공유해두면 효율적
- 그래도 모자라면 진도 빠른 수강생을 임시 도우미로 활용
- **"내 거는 안 돼요" 호소** 다수 발생: 한 명씩 다 보지 말고, 화면을 모니터에 띄워서 공통 에러 1~2개 일괄 해결
- 끝까지 배포 못한 사람: "Lovable 안에서만 동작하는 상태도 OK, Day 4에 다시 시도" 안내
- 시간 부족 시 우선순위: **회원가입 > CRUD > 배포** 순. 배포 못해도 다음 날 됨

## 예상 질문 & 대응

- **"환경 변수가 뭐예요?"** → 비밀번호 같은 거. 코드에 직접 적으면 노출되니까 따로 관리
- **"내 데이터 다른 사람이 볼 수 있나요?"** → RLS 안 켜면 그럴 수 있음. 켜는 법 시연
- **"Supabase 무료로 계속 쓸 수 있어요?"** → 일정 사용량까진 무료. 사용자 5만명 넘기 전엔 거의 무료라고 보면 됨
- **"이 사이트가 다운되면요?"** → Vercel/Supabase가 알아서 살려줌. 신경 쓸 일은 거의 없음
- **"내 도메인(.com) 연결할 수 있어요?"** → Vercel에서 가능. Day 4에 다룰 예정
