# 강의 이후 다음 단계 — AI 도구로 앱 런칭 풀 사이클

## 가장 중요한 메시지

> **이 강의는 끝이 아니라 시작입니다.**
> 4일간 만든 웹앱을 **사람들이 진짜로 쓰는 앱**, 더 나아가 **App Store / Play Store에 올라간 앱**으로 가져가는 과정을 단계별로 안내합니다.

코딩을 따로 배워야 하는 게 아니라, **지금까지 쓴 AI 도구만으로** 충분히 갈 수 있습니다. 이 페이지는 그 풀 사이클의 지도입니다.

## 전체 로드맵 한눈에

```
[ 4일 강의 끝 ]
       ↓
Phase 1 — 검증 (1~2주)
   사용자 1~5명에게 보여 주고 진짜 쓰는지 보기
       ↓
Phase 2 — 다듬기 (2~4주)
   인터뷰 받은 피드백으로 1~2개 기능 보강
       ↓
Phase 3 — 정식 웹 출시 (4~6주)
   도메인·랜딩·SEO·결제까지 정리해서 공개 출시
       ↓
Phase 4 — 모바일 앱화 (선택, 6~10주)
   PWA → Capacitor → 스토어 등록
       ↓
Phase 5 — 운영·확장 (계속)
   사용자 늘리기, 기능 보강, 수익 검증
```

각 Phase는 **이전 단계 결과를 보고 진행 여부를 결정**하면 됩니다. Phase 4(스토어 배포)는 선택입니다 — 웹만으로 잘 돌아가는 앱도 많습니다.

---

## Phase 1 — 검증 (강의 후 1~2주)

이 단계의 목표는 **앱을 더 만드는 게 아니라**, **진짜 사용자 1명**을 만드는 것입니다.

### 1주차 — 사용자 1명 확보

```
[ 목표 ]
본인 앱을 진짜로 쓰는 사람 1명 만들기

[ 방법 ]
1. 본인 SNS·블로그·카톡 단톡방에 앱 공유
2. "5분만 써 보고 피드백 주실 분" 모집
3. 1명이 가입해서 데이터 1개라도 입력하면 성공
```

### 2주차 — 그 사용자에게 인터뷰

```
[ 질문 예시 ]
- 어디서 막혔나요?
- 어떤 부분이 가장 좋았나요?
- 이 앱이 더 나아진다면 뭐가 좋아질까요?
- 친구한테 추천한다면 뭐라고 하시겠어요?
```

<Callout type="info">
**이 단계에서 앱이 별로면 멈추는 것도 정답입니다.** 사용자 1명도 안 쓰는 앱을 스토어에 올리는 건 의미가 없습니다. 다른 아이디어로 다시 4일 사이클을 돌리는 게 빠릅니다.
</Callout>

## Phase 2 — 다듬기 (2~4주)

검증에서 살아남았다면 이 단계로 옵니다.

### 핵심 작업 3가지

1. **인터뷰에서 나온 가장 큰 요청 1개**를 Lovable에 1메시지로 추가
2. **에러·버그 정리** — 사용자가 막힌 지점 우선
3. **온보딩 화면 1개** — 처음 들어온 사용자가 1분 안에 쓸 수 있게

### Lovable 한 메시지 예시

```text filename="Phase 2 보강 프롬프트"
사용자 인터뷰에서 다음 피드백을 받았어:
"[인터뷰에서 실제로 들은 한 줄]"

이걸 반영해서 다음을 한 번에 수정해 줘:
- 첫 화면에 "처음이세요?" 온보딩 카드 1개 (3단계 안내, 닫기 가능)
- "[기능명]" 버튼이 너무 작다 → 모바일에서 더 크고 진하게
- 에러 메시지가 영어로 떴음 → 모두 한국어로
```

### 사용자 5명까지 늘리기

- **디스콰이엇** ([disquiet.io](https://disquiet.io)) — 한국 메이커 커뮤니티, 빌드인퍼블릭에 적합
- **카톡 오픈채팅** — 본인 앱 타깃 주제 채팅방에서 정중히 공유
- **X / Threads** — 만드는 과정 공개하면 잠재 사용자가 따라옴

## Phase 3 — 정식 웹 출시 (4~6주)

여기까지 오면 **웹앱으로 정식 운영**할 수 있는 단계입니다.

| 항목 | 도구 / 방법 | Day |

### Lovable에 한 메시지로 분석·에러 추적 붙이기

```text filename="분석/에러 추적 추가 프롬프트"
이 앱에 다음 두 가지를 추가해 줘:
1. Vercel Analytics 또는 Plausible로 페이지뷰 추적
2. Sentry로 에러 자동 수집
   - 환경 변수로 DSN 받기
   - 익명 에러만, 개인정보는 보내지 말 것

한국 개인정보보호법 기준에 어긋나지 않게.
```

## Phase 4 — 모바일 앱화 (선택, 6~10주)

웹앱이 자리 잡았으면 **App Store / Google Play**에 올리는 걸 검토할 수 있습니다. 단, 모바일 앱은 **유지비·심사·업데이트 부담**이 추가됩니다. 사용자가 "스토어에서 다운로드받고 싶다"는 요구가 실제로 있을 때만 가는 게 좋습니다.

### 모바일화 3가지 경로 — 본인 상황에 맞게 선택

| 경로 | 난이도 | 비용 | 결과물 | 추천 시나리오 |
| --- | --- | --- | --- | --- |
| **A. PWA** | 가장 낮음 | 0원 | "홈 화면에 추가" 가능한 웹앱 | 일단 아이콘 띄워 보고 싶을 때 |
| **B. Capacitor 래핑** | 중간 | 스토어 등록비만 | 스토어에 올라가는 진짜 앱 | 웹앱 그대로 스토어에 |
| **C. 네이티브 재제작** | 높음 | 시간 비용 큼 | 푸시·카메라 등 풀 네이티브 기능 | 모바일이 본업이 될 때 |

대부분의 강의 결과물 앱은 **B(Capacitor)** 가 가장 현실적입니다.

### 경로 A — PWA (Progressive Web App)

가장 가벼운 모바일화입니다. 사용자가 사파리/크롬에서 본인 앱을 열고 **"홈 화면에 추가"** 를 누르면 아이콘이 생기고, 앱처럼 전체화면으로 실행됩니다.

```text filename="PWA 변환 프롬프트"
이 앱을 PWA로 변환해 줘.
- manifest.json 생성 (앱 이름, 아이콘, 테마 색상)
- service worker 등록 (오프라인 캐시는 최소만)
- 아이콘은 첨부한 logo.png를 512px / 192px / apple-touch-icon으로 자동 생성
- index.html에 PWA 메타 태그 추가
- 설치 가능 상태인지 Lighthouse PWA 검사 통과되는 수준으로
```

장점:
- **스토어 심사 없음**, 즉시 가능
- 비용 0원
- 업데이트는 웹 배포만 하면 자동 반영

한계:
- iOS는 푸시 알림·카메라 등 일부 기능 제한
- "스토어에서 다운로드"라는 사용자 인식이 없음

### 경로 B — Capacitor로 스토어에 올리기

웹앱을 **그대로 감싸서 네이티브 앱으로 빌드**하는 도구. [Capacitor](https://capacitorjs.com)는 Ionic이 만든 오픈소스이고, Lovable이 만든 React 앱과 자연스럽게 호환됩니다.

```
[ Capacitor 흐름 ]

Lovable에서 만든 웹앱 (React)
         ↓ npm install @capacitor/core @capacitor/cli
         ↓ npx cap init
         ↓ npx cap add ios / npx cap add android
         ↓ npx cap sync
[ iOS/Android 네이티브 프로젝트 자동 생성 ]
         ↓ Xcode (Mac 필요) / Android Studio
[ 빌드 + 스토어 업로드 ]
```

#### Capacitor 추가를 AI에게 시키기

```text filename="Capacitor 추가 프롬프트 (Lovable 또는 Cursor)"
이 React 웹앱을 Capacitor로 감싸서 iOS·Android 빌드 가능하게 해 줘.

요구사항:
- @capacitor/core, @capacitor/cli, @capacitor/ios, @capacitor/android 추가
- capacitor.config.ts 생성
  - appId: com.[본인이름].[앱이름]
  - appName: [앱 이름]
  - webDir: dist
- iOS·Android 네이티브 프로젝트 추가 명령어 README에 정리
- 빌드 후 cap sync 자동화 스크립트 (package.json scripts)
- 스플래시 화면·앱 아이콘 자리 만들어 두기 (assets/icon.png 1024x1024)
```

이후 **Mac**(iOS 빌드용) / **Windows or Mac**(Android 빌드용)에서 Xcode·Android Studio를 한 번씩 열어 스토어 업로드용 빌드를 만듭니다.

<Callout type="warning">
**iOS 빌드는 Mac이 필요합니다.** Mac이 없으면 [MacInCloud](https://www.macincloud.com) 같은 클라우드 Mac을 시간당 빌리거나, Mac 가진 지인에게 한 번 도움 받는 방법이 있습니다.
</Callout>

#### Capacitor 대안

| 도구 | 특징 |
| --- | --- |
| [Tauri](https://tauri.app) | 데스크톱 앱(Mac/Windows/Linux)으로 감쌀 때 |
| [Expo](https://expo.dev) (React Native) | 모바일이 본업이 될 거면 처음부터 Expo 추천 |
| [PWABuilder](https://www.pwabuilder.com) | PWA를 자동으로 스토어용 패키지로 변환 (가장 단순) |

### 경로 C — 네이티브 재제작 (FlutterFlow / Expo + AI)

모바일이 **본업이 될 거라면** 처음부터 모바일 네이티브로 만드는 게 결국 더 빠릅니다. AI 도구도 이미 충분히 강력합니다.

| 도구 | 비고 |
| --- | --- |
| [FlutterFlow](https://flutterflow.io) | 노코드 모바일 앱 빌더, Lovable과 비슷한 사용감 |
| [Expo](https://expo.dev) + Cursor | React Native + AI 코드 에디터, 가장 강력한 풀 네이티브 흐름 |
| [Bravo Studio](https://www.bravostudio.app) | Figma 디자인 → 앱으로 변환 |

웹과 모바일이 **둘 다** 필요한데 처음부터 React Native로 가지 않았다면, 이번 강의 결과물은 **웹**으로 두고 모바일 버전은 별도 프로젝트로 다시 만드는 것도 합리적인 선택입니다.

## App Store / Google Play 배포 실무

Phase 4 경로 B 또는 C를 선택했다면, 결국 **스토어 등록**을 해야 합니다.

### 비용·계정

| 항목 | Apple App Store | Google Play |
| --- | --- | --- |
| **개발자 계정 비용** | **$99/년** (재계약) | **$25 일회** (평생) |
| 가입 사이트 | [developer.apple.com](https://developer.apple.com) | [play.google.com/console](https://play.google.com/console) |
| 가입 시 필요한 것 | 신용카드, 본인 인증 | 신용카드, 신분증 사진 (2026 정책상 본인 확인 강화됨) |
| 가입 처리 시간 | 1~3일 | 1~2일 |

<Callout type="warning">
**개인 개발자도 사업자 등록을 권장합니다.** 한국 기준 "개인" 자격으로 등록할 수 있지만, 매출 발생 시 세무 처리가 까다롭습니다. 사업자 등록(아래 행정 섹션)을 마친 뒤 등록하는 게 깔끔합니다.
</Callout>

### 스토어 심사 흐름

```
[ Apple App Store ]

1. 빌드 업로드 (Xcode → App Store Connect)
2. 메타데이터 작성 (앱 설명, 스크린샷 6.7"/6.5"/5.5", 키워드)
3. 심사 요청 → 평균 1~3일 (변동, 첫 출시는 더 길 수 있음)
4. 승인 / 반려 (반려율 약 30% — 첫 출시 대부분 반려 한 번 거침)
5. 게시
```

```
[ Google Play ]

1. 내부 테스트 → 비공개 테스트 → 프로덕션 (단계별)
2. 메타데이터 작성 (스크린샷, 그래픽 자산, 카테고리)
3. 콘텐츠 등급 설문 + 데이터 안전 섹션
4. 심사 → 평균 1~3일
5. 게시
```

### 첫 출시 시 자주 막히는 지점 (반려 사유 Top)

| 반려 사유 | 어떻게 피하나 |
| --- | --- |
| **개인정보처리방침 누락** | 앱 안 어디서든 클릭 가능한 링크 + 스토어 메타에 별도 등록 |
| **로그인 필수인데 게스트 모드 없음** (iOS) | "둘러보기" 모드 1개 추가 또는 가입 안 해도 첫 화면 보이게 |
| **결제를 외부 링크로 처리** (iOS) | 디지털 상품은 반드시 In-App Purchase 사용 (수수료 15~30%) |
| **스크린샷·아이콘이 실제 앱과 다름** | 실제 화면 그대로 캡처 |
| **빈 화면·로딩 무한** | 네트워크 차단 환경에서도 빈 화면 안 뜨게 |
| **메타데이터에 "최고", "1위" 같은 과장** | 사실에 기반한 객관적 표현으로 |

### 스토어 메타데이터 — AI에게 시키기

```text filename="스토어 메타데이터 생성 프롬프트"
[본인 앱 PRD]
"[Day 2에 작성한 PRD 그대로 붙여넣기]"

위 PRD 기반으로 다음을 작성해 줘.

A. App Store용 (한국어 / 영어 둘 다)
- 앱 이름 (30자 이내)
- 부제 (30자 이내)
- 프로모션 텍스트 (170자)
- 설명 (4,000자, 핵심은 첫 3줄)
- 키워드 (100자, 쉼표 구분)
- "What's New" 첫 출시용 한 줄

B. Google Play용 (한국어 / 영어 둘 다)
- 앱 이름 (30자)
- 짧은 설명 (80자)
- 자세한 설명 (4,000자)
- 카테고리 추천 1개

규칙:
- "최고", "1위", "공식" 같은 심사 거부 트리거 단어는 빼기
- 비전공자가 읽어도 1초 안에 무엇 하는 앱인지 알게
- 구체적인 사용 시나리오 1개 포함
```

## Phase 5 — 운영·확장 (계속)

스토어 등록까지 마쳤으면 다음은 **숫자를 보면서 의사결정**하는 단계입니다.

### 매주 한 번씩 보면 좋은 숫자

| 지표 | 어디서 보는가 | 의미 |
| --- | --- | --- |
| 신규 가입자 수 | Lovable Cloud Auth / Users | 마케팅이 작동하는가 |
| 일일 활성 사용자 (DAU) | Plausible, GA4 | 진짜 쓰는 사람 수 |
| 다음 날 재방문율 (D1 Retention) | GA4 | 앱이 가치 있는가 (30% 넘으면 매우 좋음) |
| 결제 전환율 | Stripe / 토스 / Gumroad 대시보드 | 가격이 적절한가 |
| 에러 발생 건수 | Sentry | 어디서 사용자가 막혔는가 |

### 기능 추가 우선순위 정하기

```
[ 매주 결정 흐름 ]
1. 사용자 인터뷰 / 메시지에서 가장 자주 나온 요청 1개
2. Sentry에서 가장 많이 발생한 에러 1개
3. 결제 전환을 막는 가장 큰 장애 1개

→ 이 3개 중 1개만 골라 1주일 안에 반영
```

```
   - 디지털 상품: Apple/Google 30% 수수료
```
