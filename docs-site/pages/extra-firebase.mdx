# (번외) Supabase vs Firebase

<Callout type="info">
이 페이지는 **번외 자료**입니다. 강의 본 내용이 아니라, 강의 후 "다른 BaaS 선택지도 있나?" 궁금할 때 참고하세요.
</Callout>

## 왜 이 페이지가 있나요?

이 강의에서는 **Lovable Cloud**(엔진은 [Supabase](https://supabase.com))를 백엔드(BaaS)로 씁니다. 그런데 비전공자도 한 번쯤 들어봤을 이름이 하나 더 있어요 — **[Firebase](https://firebase.google.com)** (구글의 BaaS).

> "왜 Firebase 안 써요?"
> "둘 다 들어봤는데 뭐가 다른 거예요?"
> "내가 만들 다음 앱은 어느 쪽이 더 맞을까요?"

이런 질문에 대한 답을 한 페이지로 정리했습니다.

## 한 줄 요약

- **웹 + 비전공자 + 빠른 MVP** → **Supabase** (이 강의의 선택)
- **모바일 앱 + 푸시 알림 + 대규모 실시간** → **Firebase**
- 둘 다 **무료 한도 안에서** 시작 가능합니다.

## 핵심 차이 7가지

| 비교 항목 | Supabase | Firebase |
| --- | --- | --- |
| **데이터 구조** | SQL (엑셀 시트 같은 표) | NoSQL (문서 안에 문서) |
| **요금 방식** | DB 용량·대역폭 기준 | **읽기·쓰기 횟수당** 과금 |
| **권한 설정** | Postgres RLS — DB에 규칙 1줄 | Security Rules — 별도 문법 |
| **AI 코딩 도구 호환** | Lovable·v0·Cursor 1순위 지원 | 지원되지만 설정 더 복잡 |
| **소스 공개** | 오픈소스, 자체 호스팅 가능 | 구글 비공개 |
| **모바일 푸시 알림** | 직접 구성 필요 | **FCM 기본 제공** |
| **대규모 실시간** | 충분 | 채팅·게임 같은 영역에서 더 검증됨 |

## 항목별 자세히

### 1. 데이터 구조 — SQL vs NoSQL

가장 큰 차이입니다.

**Supabase (SQL · Postgres)**

엑셀 시트와 거의 같습니다. 행과 열로 된 표 여러 개를 두고, 표끼리 연결합니다.

```
[ users 테이블 ]              [ workouts 테이블 ]
| id | email          |       | id | user_id | 운동 | 무게 |
|----|----------------|       |----|---------|-----|-----|
| 1  | a@example.com  |       | 1  | 1       | 벤치 | 60  |
| 2  | b@example.com  |       | 2  | 1       | 스쿼트 | 80  |
                              | 3  | 2       | 데드 | 100 |
```

**Firebase (NoSQL · Firestore)**

폴더 안에 폴더를 두는 방식입니다. 표가 아니라 "문서"의 모음입니다.

```
users/
  user_1/
    email: "a@example.com"
    workouts/
      workout_1: { 운동: "벤치", 무게: 60 }
      workout_2: { 운동: "스쿼트", 무게: 80 }
  user_2/
    email: "b@example.com"
    workouts/
      workout_1: { 운동: "데드", 무게: 100 }
```

#### 비전공자 입장에서

| | Supabase (SQL) | Firebase (NoSQL) |
| --- | --- | --- |
| 직관성 | **엑셀과 같음** → 쉽게 이해 | "왜 폴더 구조?" → 새 개념 학습 필요 |
| 간단한 통계 | "이번 주 운동 합계" 같은 질문에 답하기 쉬움 | 통계용 데이터를 따로 관리해야 하는 경우 많음 |
| 표끼리 연결 | 회원-운동기록 연결이 자연스러움 | 직접 데이터를 복제해 두는 패턴이 흔함 |

### 2. 요금 — 사용량 vs 횟수

**이게 비전공자에게 가장 위험한 차이**입니다.

**Supabase**: DB 용량(MB)과 대역폭(GB)으로 과금. 무료 한도 안에서는 무엇을 얼마나 호출하든 무료.

**Firebase**: 읽기 1번, 쓰기 1번마다 카운트. 무료 한도(일 5만 건 정도)를 넘으면 **건수당 과금**.

<Callout type="warning">
**실제로 자주 일어나는 사고**: 코드에 무한 루프나 잘못된 자동 호출이 들어가면, Firebase는 하룻밤에 수십만 원이 청구된 사례가 흔합니다. 이걸 "Firebase 폭탄 청구서"라고 부르기도 해요. Supabase는 같은 실수를 해도 무료 한도 안에서 멈춥니다.
</Callout>

### 3. 권한 — RLS vs Security Rules

"본인 데이터만 보이게" 같은 규칙을 어디에 적느냐의 차이입니다.

**Supabase (RLS)**: DB 자체에 SQL 한 줄 추가.

```sql
-- 본인이 작성한 행만 조회 가능
auth.uid() = user_id
```

**Firebase (Security Rules)**: 전용 문법으로 별도 파일에 작성.

```
match /workouts/{id} {
  allow read: if request.auth.uid == resource.data.user_id;
}
```

둘 다 가능하지만, **Supabase는 우리가 어차피 SQL을 쓰고 있어서 같은 언어**로 처리 가능합니다.

### 4. AI 코딩 도구와의 궁합

이 강의가 "바이브 코딩" 강의라 이게 사실상 가장 큰 실용적 차이입니다.

| 도구 | Supabase 통합 | Firebase 통합 |
| --- | --- | --- |
| Lovable | **1순위 공식 통합 버튼** | 가능하지만 수동 설정 |
| v0 (Vercel) | 공식 가이드 풍부 | 가능 |
| Cursor / Claude Code | "Supabase 연결해 줘" 한 줄로 됨 | 추가 설명 필요 |

> **AI에게 코딩을 맡기는 시대에는, AI가 가장 잘 다루는 도구가 가장 빠릅니다.** 현재로서는 Supabase가 그 위치에 있어요.

### 5. 오픈소스 / 잠금

**Supabase**: 오픈소스. 회사가 망해도 본인 서버에 옮겨 운영 가능.

**Firebase**: 구글 비공개 서비스. 구글에 종속됩니다 (단, 구글이 망할 가능성은 거의 없죠).

장기 운영 관점에서는 Supabase가 안심되지만, **현실적으로 대부분 사용자에게는 큰 차이 없음**.

### 6. 모바일 푸시 알림

**Firebase의 강점**입니다.

- Firebase에는 **FCM(Firebase Cloud Messaging)** 이라는 푸시 알림 서비스가 기본 포함
- 모바일 앱 푸시 알림은 거의 업계 표준
- Supabase로도 가능하지만 외부 서비스(예: OneSignal)를 따로 붙여야 함

**푸시가 핵심 기능인 앱(알림 위주의 모바일 앱)이면 Firebase가 유리**합니다.

### 7. 대규모 실시간 트래픽

수만 명이 동시에 접속하는 채팅·게임·라이브 방송 같은 영역.

- Firebase는 이런 용도로 오래 쓰여 왔고 검증됨
- Supabase Realtime도 충분히 좋지만, 검증 사례는 Firebase가 더 많음

> **강의 수준의 앱(수백~수천 사용자)에서는 차이 없음.** 이건 사용자 10만 명 이상 갈 때 신경 쓸 영역입니다.

## 그럼 언제 Firebase를 골라야 하나요?

다음 중 하나라도 해당되면 Firebase 검토:

- [ ] **네이티브 모바일 앱 (iOS·Android)** 이 메인이다
- [ ] **푸시 알림**이 핵심 기능이다
- [ ] 이미 **구글 생태계** (Google Analytics, AdMob, Google Auth)를 쓰고 있다
- [ ] **수십만 명 동시 접속** 실시간 채팅·게임을 만든다

아래 경우는 그냥 Supabase 추천:

- [ ] 웹 앱이다
- [ ] 빠르게 MVP를 만들어 검증하고 싶다
- [ ] Lovable·v0 같은 AI 코딩 도구를 쓴다
- [ ] 데이터를 표 형태로 다룰 게 많다 (회원, 게시글, 주문, 기록 등)
- [ ] **요금이 예측 가능했으면 좋겠다**

## 그 외 BaaS 선택지 (참고)

세상에는 Supabase·Firebase 말고도 더 있습니다.

| 도구 | 특징 | 비고 |
| --- | --- | --- |
| [AWS Amplify](https://aws.amazon.com/amplify) | 아마존의 BaaS | 강력하지만 복잡 |
| [Appwrite](https://appwrite.io) | 오픈소스, Supabase 경쟁 | 자체 호스팅 위주 |
| [PocketBase](https://pocketbase.io) | 단일 파일로 실행되는 BaaS | 1인 프로젝트용 |
| [Convex](https://convex.dev) | 신생, 함수형 데이터 | AI 연동 강함 |

비전공자 입장에서는 **Supabase 또는 Firebase 둘 중 하나**로 시작하는 게 정보·튜토리얼이 가장 많아 추천합니다.
