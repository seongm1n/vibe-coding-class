# PRD 사고법 — 좋은 프롬프트는 좋은 요구사항이다

## "PRD"가 뭔가요?

**PRD (Product Requirements Document)** 는 우리말로 옮기면 "제품 요구사항 문서"입니다. 회사에서 새 제품을 만들기 전에 작성하는 1~2장짜리 문서입니다.

거창하게 들리지만, 본질은 다음 4가지를 정리하는 것입니다.

> **누가, 왜, 무엇을, 어떻게**

이걸 정리해 두면 누가 만들어도 비슷한 결과물이 나옵니다. AI에게 시킬 때도 마찬가지입니다.

<Callout type="info">
**비전공자도 PRD를 모르는 게 아니라, 이름만 모릅니다.**
어머니가 자녀에게 "내일 도시락 싸 줄 건데, 너 좋아하는 반찬 3개 정해서 알려 줘. 매운 건 빼고"라고 말하는 것도 PRD의 한 형태입니다.
</Callout>

## PRD의 4가지 핵심 요소

### 1. 누가 (타깃 사용자)

이 앱을 누가 쓸 것인가? **한 명을 구체적으로 묘사**할수록 좋습니다.

```text
❌ 나쁜 예: "20~40대 직장인"
✅ 좋은 예: "30대 중반, 세종시 정부청사에 근무하고, 점심 시간에 매번 어디 갈지 고민하는 직장인"
```

### 2. 왜 (해결하려는 문제)

그 사람이 지금 어떤 문제를 겪고 있는가?

```text
❌ 나쁜 예: "식당 정보가 부족해서"
✅ 좋은 예: "점심 12시쯤 동료들과 갈 식당을 골라야 하는데, 매번 같은 곳만 가게 되고
            새 곳을 찾으려면 네이버지도·인스타·블로그를 다 뒤져야 함"
```

### 3. 무엇을 (핵심 기능)

이 앱이 어떤 기능을 제공하는가? **3개 정도가 적당합니다**.

```text
✅ 좋은 예:
1) 점심 시간에 맞춰 오늘의 추천 식당 3곳 제시
2) 거리·가격대·메뉴 종류로 필터
3) "오늘 여기" 버튼으로 동료들과 빠르게 합의
```

### 4. 어떻게 (화면 구성)

가장 중요한 화면 2~3개에 무엇이 보이는가?

```text
✅ 좋은 예:
- 메인 화면: 추천 카드 3개(이름·거리·메뉴) + 필터 버튼
- 상세 화면: 식당 정보 + 지도 + 후기
- 결과 화면: 동료들과 합의된 식당 + 가는 길
```

## 프롬프트 작성의 3가지 원칙

PRD를 다 적었다 해도, 그걸 그대로 AI에게 던지기 전에 **컨텍스트·목표·제약** 세 가지가 모두 들어갔는지 확인하세요.

### 1. 컨텍스트 (Context)

본인이 누구인지, 어떤 상황인지 알려줍니다.

```text
"나는 30명 규모 헬스장을 운영하는 사장이다.
회원들이 매번 카운터에 와서 출석 체크를 하는데, 카운터가 비는 시간이 많아서 셀프 체크가 필요하다."
```

### 2. 목표 (Goal)

무엇을 만들고 싶은지 명확히 합니다.

```text
"회원이 본인 휴대폰으로 QR 코드를 스캔하면 자동으로 출석이 기록되는 화면이 필요하다."
```

### 3. 제약 (Constraints)

어떤 조건을 지켜야 하는지 알려줍니다.

```text
"- 모바일 화면에 최적화 (가로 350px 정도)
- 글자는 크게 (어르신 회원도 있음)
- 한국어로
- 색상은 헬스장 로고 색(짙은 주황)에 맞춰
- 회원이 1초 안에 어떻게 해야 할지 알 수 있게 단순하게"
```

## 다 같이 따라 하기 — PRD 작성

이 챕터는 **다 같이 샘플 PRD 한 번 만들어 보기 → 본인 PRD 작성 → AI에게 자체 검증** 흐름으로 갑니다.

### 사전 준비

브라우저 탭 2개를 엽니다.

| 탭 | 사이트 | 용도 |
| --- | --- | --- |
| 1 | [Claude](https://claude.ai) 또는 [ChatGPT](https://chatgpt.com) | PRD 생성 |
| 2 | 메모 앱 (메모장 / Notion / 텍스트 편집기) | PRD 결과 저장 |

<Callout type="info">
**PRD 작성에는 Claude가 약간 더 잘 맞는 편**입니다. 한국어 톤이 자연스럽고, 긴 글을 정리할 때 ChatGPT보다 안정적입니다. 둘 다 가능하니 본인이 익숙한 쪽으로.
</Callout>

### 다 같이 — ChatGPT로 샘플 PRD 만들기

본인 앱이 아니라 **공통 샘플 컨셉**으로 PRD를 한 번 같이 뽑아 봅니다. 흐름과 결과 형태만 머리에 넣는 게 목적입니다. 본인 앱 PRD는 다음 단계에서 만듭니다.

```
[ 다 같이 ① ] 아래 프롬프트 복사 → ChatGPT(또는 Claude)에 붙여넣기
[ 다 같이 ② ] Enter
   ↓ 1~2분 대기
[ 다 같이 ③ ] 결과 위에서 아래로 한 번 읽기
[ 다 같이 ④ ] 본인이 자문 — "이 정도면 화면 만들 수 있겠어 보이는가?"
```

다음 프롬프트를 그대로 사용합니다.

```text filename="ChatGPT용 PRD 생성 프롬프트"
나는 [본인 컨셉]을 만들고 싶다.
비개발자도 이해할 수 있는 1페이지 PRD를 만들어줘.

다음 형식으로:
1. 타겟 사용자 (한 명을 구체적으로)
2. 해결하려는 문제 (그 사람이 지금 겪고 있는 불편)
3. 핵심 기능 3개
4. 가장 중요한 화면 2~3개의 구성 요소
5. 디자인 톤 (색상·분위기)
6. 모바일/데스크톱 어디 우선인지

너무 거창하게 만들지 말고, 4일 강의에서 비전공자가 만들 수 있을 만큼 단순하게.
```

ChatGPT 답이 나오면 함께 읽으면서 "이 정도면 AI가 화면 만들 수 있겠다"는 감을 잡습니다.

<Callout type="info">
**팁: Claude를 사용해도 좋습니다.** Claude는 한국어 글의 톤이 좀 더 자연스러운 편입니다. 같은 프롬프트를 ChatGPT와 Claude 양쪽에 던지고 결과가 더 나은 걸 고르세요.
</Callout>

### 본인 PRD 작성 — 단계별 흐름

이제 본인 앱으로 직접 PRD를 만듭니다. 다음 4단계를 차례로 따라가세요.

#### Step 1 — 어제 메모를 한 단락으로 풀기

Day 1 숙제로 가져온 3줄 메모를 **하나의 단락(5~7문장)** 으로 늘리세요.

```text filename="1단락 PRD 시드 — 메모장에 적기"
나는 [본인 상황·직업].
[본인 또는 주변 사람이 겪는 불편] 한 줄.
[지금은 이 문제를 어떻게 풀고 있는지] 한 줄.
[그게 왜 별로인지] 한 줄.
[내가 만들 앱이 어떻게 다르게 풀어주는지] 한 줄.
[누가 이걸 가장 좋아할지] 한 줄.
```

본인 단락이 5문장 이상이 되면 OK. **3문장 이하면 다시 짜세요** — AI가 PRD를 제대로 만들 정보가 부족합니다.

#### Step 2 — Claude/ChatGPT에 PRD 생성 시키기

방금 적은 1단락을 다음 프롬프트의 `[1단계 단락]` 자리에 붙여 넣고, 챗봇에 던집니다.

```text filename="PRD 생성 프롬프트 — 본인 단락만 채워서 그대로 사용"
다음은 내가 만들고 싶은 앱의 1단락 설명이다.

"[1단계에서 적은 단락 그대로 붙여넣기]"

이 설명을 1페이지 PRD로 정리해 줘. 비개발자도 이해할 수 있게.

다음 형식으로:
1. 타겟 사용자 (한 명을 구체적으로 — 나이/직업/상황)
2. 해결하려는 문제 (그 사람이 지금 겪는 불편을 구체적으로)
3. 핵심 기능 3개 (4일 안에 비전공자가 만들 수 있을 만큼 단순하게)
4. 가장 중요한 화면 2~3개의 구성 요소
   - 메인 화면에 무엇이 보이는가
   - 사용자가 가장 자주 누르는 버튼은 무엇인가
5. 디자인 톤 (색상·분위기 한 문장)
6. 모바일/데스크톱 우선

추가 조건:
- 노코드 도구(Lovable)로 만들 거라 너무 복잡한 기능은 빼 줘
- 백엔드(데이터 저장)는 다음 날에 추가할 거니까, 오늘은 화면 위주로
- 한국어로
```

답이 1~2분 안에 나옵니다. **메모장에 그대로 복사** 해 두세요.

#### Step 3 — AI 답을 본인 말로 다시 손질

AI가 만든 PRD를 그대로 쓰지 마세요. **본인 색깔이 들어가야 좋은 PRD** 입니다. 다음 항목을 하나씩 짚으면서 손질하세요.

| 점검 | 무엇을 보는가 |
| --- | --- |
| 타깃 사용자 | 진짜 본인이 알고 있는 사람과 비슷한가? 아니면 일반론인가? |
| 문제 | 본인이 진짜 느낀 불편인가? AI가 지어낸 게 아닌가? |
| 핵심 기능 3개 | 정말 다 필요한가? 1개는 빼도 되지 않나? |
| 화면 구성 | 머릿속에 그림이 그려지는가? 안 그려지면 더 구체적으로 |
| 디자인 톤 | 본인 취향이 들어갔나? "모던"같이 두루뭉술한 단어 빼기 |

**불필요한 기능은 과감히 삭제**하세요. PRD에 적힌 모든 기능이 결국 만들 항목이 됩니다.

#### Step 4 — Lovable용 첫 프롬프트로 변환

손질된 PRD를 **Lovable에 던질 1개 프롬프트** 로 바꿉니다. 다음 챕터([내 앱 컨셉 정의서](/day2/concept))에 변환 템플릿이 있습니다. 거기로 이동하세요.

### AI 자체 검증

PRD를 다 적었으면, 같은 챗봇에 다음 프롬프트로 다시 던지세요. **본인이 못 보는 모호한 부분을 AI가 짚어 줍니다.**

```text filename="PRD 자체 검증 프롬프트"
다음은 내가 작성한 PRD다.

"[방금 손질한 PRD 그대로 붙여넣기]"

이 PRD를 보고 다음을 알려줘:
1. 화면 만들 때 헷갈릴 만한 모호한 부분 3개
2. 빠진 정보 (있다면 무엇이 빠졌는지)
3. 너무 복잡해서 4일 안에 못 끝낼 것 같은 기능이 있다면 그게 뭔지
4. 한 문장으로 "이 앱은 OOO한 사람을 위해 OOO을 해주는 앱이다" 요약해줘

비개발자도 이해할 수 있게 답해줘.
```

AI가 짚어 준 모호한 부분을 PRD에 보강하세요. 이걸 1~2번 반복하면 PRD 품질이 눈에 띄게 올라갑니다.

<Callout type="info">
**핵심**: 본인이 머릿속에만 갖고 있는 정보가 글에 안 적혀 있으면, 그게 곧 **AI가 화면을 만들 때 흔들리는 지점**입니다. AI에게 PRD를 검증시키는 건 "내 PRD가 글자만 보고도 이해되는가"를 확인하는 가장 빠른 방법입니다.
</Callout>

## 좋은 PRD vs 나쁜 PRD — 한눈에 비교

### 나쁜 PRD

```text filename="나쁜 PRD"
운동 기록 앱
- 운동 기록할 수 있고
- 친구랑 공유할 수도 있고
- 예쁘게 만들어 줘
```

너무 추상적입니다. AI가 무엇을 만들지 100가지 해석이 가능합니다.

### 보통 PRD

```text filename="보통 PRD"
운동 기록 앱
- 타겟: 헬스장 다니는 직장인
- 기능: 운동 종류, 횟수, 무게 기록
- 친구한테 보여 주는 기능
- 모던한 디자인
```

기본은 갖췄지만, 화면 구성과 디자인 톤이 모호합니다.

### 좋은 PRD

```text filename="좋은 PRD"
운동 기록 일기 앱

[타겟 사용자]
- 30대 직장인, 헬스장 주 3회 다님
- 운동 기록을 노트에 적었지만 잃어버리기 일쑤
- 친구랑 운동 일기 공유하면서 동기 부여 받고 싶음

[문제]
- 노트에 적으면 잃어버리고, 엑셀에 적자니 번거로움
- 헬스장 앱들은 너무 복잡해서 쓰기 귀찮음

[핵심 기능 3개]
1) 운동 1개를 30초 안에 기록 (종목 / 무게 / 횟수)
2) 이번 주 기록을 한눈에 보기
3) 친구 1명에게 이번 주 요약을 카톡으로 공유

[화면 구성]
- 메인 화면: 오늘의 기록 입력 (종목 드롭다운, 숫자 입력 2개, 저장 버튼)
- 기록 보기: 이번 주 운동 카드 7개 (요일별)
- 공유 화면: 이번 주 요약 + 카톡 보내기 버튼

[디자인 톤]
- 미니멀, 깔끔, 둥근 모서리
- 색상: 흰색 배경 + 짙은 주황(#c8401a) 강조
- 모바일 우선 (가로 폭 360~400px)

[기타]
- 한국어
- 처음 쓰는 사용자도 화면 보고 바로 알 수 있어야 함
```

이렇게 적으면 AI가 어디서 흔들리지 않고 일관된 결과를 줍니다.

## 좋은 PRD를 위한 체크리스트

PRD를 다 쓴 뒤 다음을 확인하세요.

- [ ] 타깃 사용자를 **한 명**을 구체적으로 묘사했는가
- [ ] 그 사람이 지금 겪는 불편이 적혀 있는가
- [ ] 핵심 기능이 **3개 이내**로 정리됐는가 (4일 안에 가능한 양)
- [ ] 가장 중요한 화면 2~3개의 구성 요소가 적혀 있는가
- [ ] 디자인 톤(색·분위기)이 한 줄이라도 적혀 있는가
- [ ] 모바일/데스크톱 우선 순위가 적혀 있는가

## 자주 하는 실수

<Callout type="warning">
**실수 1: 기능을 너무 많이 적음**
"이것도, 저것도, 이것도..." 식으로 5~10개 적으면 4일 안에 못 끝냅니다. **3개로 줄이세요.** 나머지는 "다음 버전" 폴더에 따로 적어 두세요.
</Callout>

<Callout type="warning">
**실수 2: 타깃을 "20~50대 모든 사람"으로 잡음**
모두를 위한 앱은 아무도 안 씁니다. **한 명**을 구체적으로 묘사하세요. 그 한 명이 좋아하면 그 주변 사람들도 따라옵니다.
</Callout>

<Callout type="warning">
**실수 3: PRD 단계에서 시간을 다 씀**
PRD는 만들면서 계속 다듬으면 됩니다. **60분 이상 PRD에 쓰지 마세요.** 일단 부족해도 Lovable로 넘어가서 화면이 나오면 그걸 보면서 PRD를 다시 다듬는 게 빠릅니다.
</Callout>
