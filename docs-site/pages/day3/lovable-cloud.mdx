# Lovable Cloud로 백엔드 켜기

## 이 챕터의 목표

Lovable이 만든 화면(프론트엔드)에 **백엔드 + 데이터베이스를 붙입니다**. 옛날엔 Supabase에 직접 가입해 프로젝트를 만들고 키를 복사해야 했지만, **이제는 Lovable Cloud 한 번 켜면 끝**입니다.

이 챕터가 끝나면 본인 프로젝트에 다음이 켜져 있습니다.

1. **Database** — 데이터를 저장할 표(테이블)들이 들어갈 곳
2. **Authentication** — 회원가입·로그인
3. **Storage** — 파일·이미지 보관소
4. **Edge Functions** — 복잡한 처리가 필요할 때 (이번 강의에선 거의 안 씀)

## 잠깐 — Lovable Cloud가 뭐야?

**Lovable Cloud**는 Lovable이 자동으로 띄워 주는 **풀스택 백엔드 호스팅**입니다. 내부적으로는 [Supabase](https://supabase.com) 엔진을 사용합니다. 즉, 앞 챕터([프론트·백엔드·DB 개념](/day3/concepts))에서 본 "BaaS"가 Lovable 안에 통합돼 있는 셈입니다.

| 옛날 방식 (Lovable Cloud 등장 전) | 지금 (Lovable Cloud) |
| --- | --- |
| 1. Supabase 가입 | (없음) |
| 2. 새 프로젝트 만들기 | (없음) |
| 3. URL과 API 키 복사 | (없음) |
| 4. Lovable 환경 변수에 붙여넣기 | (없음) |
| 5. 코드에 연결 요청 | **클릭 한 번 + 채팅 한 번** |

수강생 입장에서는 **단계 4개가 사라진 것**이 가장 큰 차이입니다.

<Callout type="info">
**그래도 Supabase 개념은 유효합니다.** Lovable Cloud의 엔진이 Supabase라서, 앞 챕터에서 본 "테이블 = 엑셀 시트", "RLS = 본인 데이터만 보이게" 같은 개념이 그대로 적용됩니다. UI만 Lovable 안에 통합돼 있을 뿐입니다.
</Callout>

## 무료로 어디까지 되나요?

Lovable의 가격은 **두 가지 축**으로 따로 청구됩니다. 헷갈리니 분리해서 봅시다.

### 축 1 — 메시지(크레딧): 채팅으로 시키는 작업

채팅창에 "이 기능 추가해 줘" 같은 요청을 던질 때 소비되는 크레딧.

| 플랜 | 가격 | 메시지 한도 |
| --- | --- | --- |
| **Free** | $0 | **하루 5 / 월 30** |
| **Pro** | $25/월 (연간 $21/월) | 월 100 + 하루 5 (최대 150) |
| **Business** | $50/월 (연간 $42/월) | 월 100 + 팀 기능 |

소비 규칙:

- **Plan 모드**: 메시지 1번 = 크레딧 1개
- **Agent 모드**: 작업 복잡도에 따라 가변 (단순 작업은 1개, 복잡한 멀티스텝은 더 많이)
- **"Try to fix" 버튼은 무료** (크레딧 안 깎임)

### 축 2 — Cloud + AI 사용량: 백엔드·AI 기능

DB·Auth·Storage·Edge Functions·AI 기능을 쓸 때 들어가는 사용량. **메시지 크레딧과 완전히 별개** 청구.

| 플랜 | Cloud 무료 한도 | AI 무료 한도 |
| --- | --- | --- |
| **Free** | **월 $25** (워크스페이스당) | **월 $1** |
| **Pro 이상** | 월 $25 포함 | 월 $1 포함 |

이 한도를 넘으면 Free는 차단, Pro 이상은 사용량 기반 추가 청구.

<Callout type="warning">
**한시 프로모션**: 무료 플랜의 월 $25 Cloud + $1 AI 무료 한도는 **2026년 5월까지** 한시 적용. 이후 정책이 바뀔 수 있어요. 강의 후 계속 쓸 계획이라면 [공식 가격 페이지](https://lovable.dev/pricing)에서 그 시점 정책을 다시 확인하세요.
</Callout>

### Free 플랜의 다른 한도

- **서브도메인 5개** (배포된 `xxx.lovable.app` 5개까지)
- **협업자 20명** (한 워크스페이스 안)
- **Lovable 배지 제거 불가** (Pro부터 가능)
- **커스텀 도메인 연결 불가** (Pro부터 가능)

### 강의 수강생 입장 — 실질 비용

| 시나리오 | 예상 비용 |
| --- | --- |
| 강의 기간 + 친구 1~2명에게 보여 주기 | **$0** (메시지·Cloud 한도 안) |
| 강의 후 SNS 공유 → 친구 수십 명 가입 | **$0~몇 달러** (대부분 한도 안) |
| 본격 운영 → 사용자 수백~수천 | Pro $25/월 검토 |
| 팀 단위 협업 | Business $50/월 검토 |

**거의 모든 수강생이 강의 기간 동안 0원**으로 끝납니다. 다만 두 가지를 미리 알아 두세요.

1. **하루 5메시지 한도** — 그래서 이 강의의 프롬프트는 "한 번에 모든 요구사항 + 화면 + 보안" 까지 다 담는 식입니다. 메시지를 쪼개면 금방 한도 도달.
2. **2026년 5월 이후 정책 변경 가능** — 그때 가서 [Lovable Pricing](https://lovable.dev/pricing)을 다시 확인.

## 단계 1 — Cloud 탭 열기

Lovable 에디터 화면 상단에 **Preview**(미리보기) 탭이 있습니다. 그 **바로 옆 `+` 버튼**을 눌러서 새 탭을 추가합니다.

```
[ Code ]  [ Preview ]  [ + ]   ← 이 + 버튼 클릭
                         ↓
              [ Cloud ] 탭 추가됨
```

`+` 버튼을 누르면 추가 가능한 탭 목록이 뜹니다. **"Cloud"** 를 선택하세요.

<Callout type="info">
**탭 이름이 다를 수 있음**: Lovable 업데이트에 따라 "Cloud" 대신 "Backend" 또는 다른 이름으로 표시될 수 있습니다. 어쨌든 데이터베이스/인증을 관리하는 탭입니다.
</Callout>

## 단계 2 — Cloud 탭 둘러보기

Cloud 탭이 열리면 좌측에 다음 메뉴가 보입니다.

| 메뉴 | 무엇을 보는 곳 |
| --- | --- |
| **Database / Tables** | 데이터가 저장되는 표(테이블) 목록. 지금은 비어 있음 |
| **Auth / Users** | 회원가입한 사용자 목록. 지금은 비어 있음 |
| **Storage** | 파일·이미지 저장소. 지금은 비어 있음 |
| **Logs** | 백엔드에서 일어난 일 기록 (에러 디버깅에 유용) |

지금은 **모든 탭이 비어 있는 게 정상**입니다. 다음 챕터에서 회원가입을 붙이면 Auth/Users에 사용자가 1명 등장합니다.

## 단계 3 — 채팅에 "백엔드 연결" 한 번 시키기

Cloud 탭만 열어 두는 것으론 코드와 자동 연결되지 않습니다. 채팅창에 한 번 시켜 주세요.

```text filename="✅ Cloud 연결 프롬프트"
이 프로젝트에 Lovable Cloud의 백엔드(데이터베이스 + 인증)를 연결해 줘.

- supabase 클라이언트가 Cloud 인스턴스에 자동 연결되도록 설정
- 환경 변수는 .env에서 자동 주입되게
- src/integrations/ 폴더에 Cloud 연결 코드 생성
```

Lovable이 자동으로 다음을 처리해 줍니다.

- `src/integrations/supabase/client.ts` 같은 파일 자동 생성
- `.env`에 `VITE_SUPABASE_URL`, `VITE_SUPABASE_PUBLISHABLE_KEY` 자동 등록
- Cloud 인스턴스의 URL/키를 자동 주입

<Callout type="info">
**환경 변수를 본인이 직접 입력할 필요 없음**. Lovable Cloud가 자동으로 만들어 주는 백엔드의 키는 Lovable이 알아서 코드에 주입합니다.
</Callout>

## 단계 4 — 연결 확인

Lovable이 작업을 끝냈다고 답하면 다음 두 가지로 확인합니다.

### 확인 1 — 코드에 연결 파일이 생겼는지

Code 탭에서 다음 파일이 보이면 OK:

```
src/
└── integrations/
    └── supabase/
        ├── client.ts      ← 자동 생성됨
        └── types.ts       ← 자동 생성됨
```

`client.ts`를 클릭해서 열어 보면 `import.meta.env.VITE_SUPABASE_URL` 같은 코드가 보일 겁니다. 이게 **환경 변수가 정상적으로 연결됐다**는 신호입니다.

### 확인 2 — Cloud 탭에 메뉴가 살아 있는지

Cloud 탭으로 돌아가서 **Database**, **Auth**, **Storage** 메뉴가 클릭 가능한 상태인지 확인. 클릭했을 때 빈 표가 보이면 정상.

## 자주 하는 실수 / 헷갈리는 점

### "왜 Supabase에 직접 가입 안 해요?"

**Lovable Cloud가 Supabase 인스턴스를 자동으로 만들어 줍니다.** 본인이 따로 supabase.com에 가입할 필요가 없어요. 단, 진짜로 본인이 Supabase를 소유하고 싶다면 [(번외) 직접 Supabase·Vercel 운영하기](/day3/advanced-supabase-vercel)를 참고하세요.

### "그럼 데이터는 누구 거예요?"

데이터는 **Lovable Cloud가 관리하는 Supabase 인스턴스**에 들어갑니다. 본인 Lovable 계정에 묶여 있어서, Lovable에서 프로젝트를 지우면 데이터도 같이 사라집니다.

장기 운영 시:

- 본인이 Supabase 직접 운영하면 → 어디든 옮길 수 있음
- Lovable Cloud 사용 → Lovable에 묶여 있음 (편의 ↔ 자유의 트레이드오프)

수강 단계에선 Lovable Cloud가 훨씬 빠르고 편합니다. 운영 단계 가서 필요하면 그때 Supabase 직접 마이그레이션을 검토해도 됩니다.

### "환경 변수는 그래도 알아 둬야 하나요?"

네. **개념은 이해해 두세요.** 코드에 비밀 키를 직접 적지 않고 `.env` 파일에서 가져온다는 원리는 동일합니다. Lovable Cloud가 자동화해 줄 뿐이지, 키 자체는 여전히 환경 변수로 관리되고 있어요.
