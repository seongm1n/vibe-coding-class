# 프론트·백엔드·DB 개념

## 이 챕터의 목표

[앞 챕터](/day3/decompose)에서 본인 앱을 프론트·백엔드·DB로 직접 쪼개 봤습니다. 이번 챕터에서는 그 위에 얹을 **추가 개념들**(BaaS, 환경 변수, RLS, 배포)을 비유와 그림으로 정리합니다. **코드를 보지 않아도 됩니다.**

## 식당 비유로 이해하기

가장 직관적인 비유입니다.

```
┌──────────────────────────────────────────────┐
│  프론트엔드      |   백엔드      |   DB        │
│  (홀)           |   (주방)      |   (냉장고)  │
│                 |              |              │
│  - 메뉴판       |  - 주문 받음  |  - 재료 보관 │
│  - 손님 응대    |  - 요리       |  - 식재료     │
│  - 결제 받음    |  - 음식 내줌  |    재고 관리  │
└──────────────────────────────────────────────┘
```

| 영역 | 식당으로 치면 | 우리 앱에서 | 어떤 도구 |
| --- | --- | --- | --- |
| **프론트엔드** | 손님이 보는 홀 | 사용자가 보는 화면 | Lovable로 만든 것 |
| **백엔드** | 주방 | 데이터 처리·로그인 | Lovable Cloud가 알아서 함 |
| **데이터베이스(DB)** | 냉장고 | 데이터가 쌓이는 곳 | Lovable Cloud 안에 들어 있음 (엔진은 Supabase) |

## 어제(Day 2) 만든 화면의 한계

어제 화면을 만들고 운동 기록을 입력하셨다면, **새로고침을 누르는 순간 사라졌을** 겁니다. 왜일까요?

```
현재: 화면(프론트엔드)만 있음
       ↓
입력한 데이터는 임시 메모리에만 있음
       ↓
새로고침 = 메모리 초기화 = 데이터 사라짐
```

이걸 해결하려면 **백엔드와 DB**가 필요합니다.

## BaaS — 백엔드를 빌려 쓰기

### "BaaS"가 뭔가요?

**BaaS (Backend as a Service)** 는 우리말로 옮기면 "서비스로 제공되는 백엔드"입니다. 백엔드를 직접 만들지 않고 **빌려 쓰는** 개념입니다.

비유하자면:

| 방식 | 비유 |
| --- | --- |
| 직접 백엔드 만들기 | 식당 차릴 때 주방을 처음부터 짓기 (몇 주~몇 달) |
| BaaS 사용 | **풀옵션 주방을 임대**하기 (몇 분 만에 사용 가능) |

이 강의에서는 **Lovable Cloud**(엔진은 [Supabase](https://supabase.com))를 씁니다. Lovable이 BaaS를 자기 안에 통합해 둬서, 본인이 따로 가입할 필요 없이 **Cloud 탭 한 번 켜면** 백엔드가 준비됩니다.

### Lovable Cloud가 해 주는 일

Lovable Cloud 안에는 다음이 다 들어 있습니다.

1. **Authentication (인증)** — 회원가입·로그인
2. **Database (데이터베이스)** — 데이터를 표(table) 형태로 저장
3. **Storage (스토리지)** — 파일·이미지 저장
4. **Edge Functions** — 더 복잡한 처리가 필요할 때
5. **AI 기능** — 요약·챗봇·감정 분석 등

오늘 우리는 **1, 2번만** 씁니다. 추후에 본인 Supabase로 직접 운영하고 싶다면 [(번외) 직접 Supabase·Vercel 운영하기](/day3/advanced-supabase-vercel)를 참고하세요.

## 데이터베이스 = 엑셀 시트

처음에 데이터베이스를 어렵게 생각하실 수 있는데, 사실 **엑셀과 거의 같습니다**.

```
[ workouts 테이블 ]

| id | user_id | date       | exercise | weight | reps |
|----|---------|------------|----------|--------|------|
| 1  | u_001   | 2026-04-29 | 벤치프레스 | 60     | 10   |
| 2  | u_001   | 2026-04-29 | 스쿼트     | 80     | 8    |
| 3  | u_002   | 2026-04-28 | 데드리프트 | 100    | 5    |
```

| 엑셀 용어 | DB 용어 |
| --- | --- |
| 시트 (Sheet) | 테이블 (Table) |
| 행 (Row) | 행 (Row) |
| 열 (Column) | 열 (Column) |
| 셀 (Cell) | 셀 (Cell) |

차이점은:

- DB는 **수십만 건도 빠르게** 검색·정렬할 수 있습니다.
- DB는 **여러 사람이 동시에** 읽고 써도 문제없습니다.
- DB는 **권한 설정**이 가능합니다 (내 데이터만 보이게).

## 환경 변수 — 비밀 키 관리 (개념만)

### 왜 환경 변수가 필요한가요?

백엔드에 연결하려면 **API 키**가 필요합니다. 이건 일종의 비밀번호입니다.

```
나쁜 방식 (하지 말 것):
코드 안에 "abc123secretkey" 라고 직접 적기
   ↓
GitHub에 코드 올리면 누구나 볼 수 있음
   ↓
나쁜 사람이 내 키로 무한정 사용 → 나에게 요금 청구
```

```
좋은 방식 (환경 변수):
코드에는 "키는 환경 변수에서 가져와"라고만 적음
   ↓
키 자체는 .env 파일 또는 호스팅 설정에 따로 저장
   ↓
GitHub에 올라가는 코드에는 키가 없음 → 안전
```

### Lovable Cloud에서는 자동 처리

Lovable Cloud를 쓰면 **본인이 키를 직접 입력할 일이 거의 없습니다.** Lovable이 `.env` 파일에 `VITE_SUPABASE_URL` 같은 값을 자동으로 채워 주고, 코드도 그 값을 자동으로 읽도록 작성해 줍니다.

직접 운영(번외 챕터)으로 갈 때만 본인이 환경 변수 이름·값을 의식해서 다루게 됩니다.

### 환경 변수 이름 규칙 (참고)

이름 앞에 `NEXT_PUBLIC_` 또는 `VITE_`가 붙으면 **브라우저(프론트엔드)에서 사용 가능한 키**입니다. 이름 앞에 아무 접두어가 없으면 **서버에서만 써야 하는 키**입니다 (예: Supabase의 Secret key).

```
VITE_SUPABASE_URL = https://xxxxx.supabase.co
VITE_SUPABASE_PUBLISHABLE_KEY = eyJxxxxx...
```

<Callout type="warning">
**Publishable key(=구 anon key)는 공개돼도 비교적 안전**합니다. 단, **Secret key(=구 service_role key)** 는 절대 공개되면 안 됩니다. 이름이 헷갈릴 때는 항상 **Publishable key** 쪽을 쓰세요.
</Callout>

## RLS — Row Level Security

### "RLS"가 뭔가요?

**Row Level Security (행 단위 보안)** 는 우리말로 옮기면 "데이터의 줄 한 줄마다 누가 볼 수 있는지 정하는 규칙"입니다.

비유하자면:

| 상황 | 설명 |
| --- | --- |
| RLS 꺼져 있음 | 도서관에서 누구나 모든 책을 가져갈 수 있음 |
| RLS 켜져 있음 | "본인이 빌린 책만 빌려 갈 수 있음" 같은 규칙이 있음 |

### 왜 중요한가요?

RLS를 안 켜면 다음 일이 일어날 수 있습니다.

- 내가 입력한 운동 기록을 **다른 사용자가 볼 수 있음**
- 또는 다른 사용자의 기록을 **내가 수정/삭제할 수 있음**

이건 큰 보안 문제입니다.

### Lovable Cloud에서는 어떻게 하나요?

다행히 Lovable Cloud는 **새 테이블을 만들면 RLS가 기본으로 켜져** 있습니다. 우리는 "본인 데이터만 볼 수 있다"는 규칙(Policy) 1줄만 추가하면 됩니다 — 그것도 채팅에 한 번 시키면 자동으로 들어가요.

이건 [CRUD 챕터](/day3/crud)에서 다 같이 따라 하면서 직접 켭니다. 지금은 **개념만 알고 있으면 충분**합니다.

## 배포 = 인터넷에 띄우기

### 지금 상태

```
Lovable 미리보기 안에서만 동작
   ↓
URL이 있긴 하지만 "에디터의 미리보기 URL"
   ↓
세션이 끊기면 일시적으로 안 보일 수 있음
```

### 배포 후 상태

```
Lovable의 Publish 버튼 클릭
   ↓
1~2분 빌드
   ↓
진짜 도메인 (예: my-app.lovable.app)
   ↓
누구나 접속 가능, 24시간 살아 있음
```

### 무료로 어디까지 되나요?

Lovable Free 플랜 안에서 다음을 제공합니다.

- 무료 도메인 `xxx.lovable.app`
- 자동 HTTPS (https:// 보안 연결)
- 워크스페이스당 월 $25 어치 Cloud 사용량 (백엔드까지 포함)
- 커스텀 도메인 연결 가능

큰 회사가 유료 플랜으로 돈을 벌고, 작은 사용자는 무료로 쓰는 구조입니다.

> Vercel·Netlify·Cloudflare Pages 같은 외부 호스팅을 쓰고 싶다면 [(번외) 직접 Supabase·Vercel 운영하기](/day3/advanced-supabase-vercel) 참고.

## 자주 묻는 질문

### Q. "코드를 봐야 하나요?"

**살짝만** 봅니다. 환경 변수 이름이 맞는지, 키 값을 어디 넣는지 정도만 확인합니다. 코드 한 줄도 직접 적을 필요는 없습니다.

### Q. "Lovable Cloud 무료로 계속 쓸 수 있어요?"

워크스페이스당 월 $25 어치까지 무료입니다 (2026년 5월까지 한시 적용, 이후 정책 변경 가능). 강의 데모 + 친구 가입 수준이면 0원으로 끝납니다.

### Q. "이 사이트가 다운되면요?"

Lovable의 호스팅이 알아서 살려 줍니다. 신경 쓸 일은 거의 없습니다. 클라우드 인프라 위에서 돌아가서 안정적입니다.

### Q. "내 도메인(.com)을 연결할 수 있어요?"

네, Lovable의 Publish 패널에서 가능합니다. 가비아·Cloudflare 같은 곳에서 도메인을 사서 (연 $10~15 정도) 연결합니다. 이 부분은 [Day 4](/day4)에서 짧게 다룹니다.
