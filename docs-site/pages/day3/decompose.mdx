# 내 앱을 3등분 해보기

## 이 챕터의 목표

- 프론트엔드 / 백엔드 / 데이터베이스가 각각 무엇을 하는지 다시 정리합니다.
- **본인이 어제까지 만든 앱**을 직접 3개 영역으로 쪼개 봅니다.
- Day 3 내내 우리가 어디를 만지는지 머릿속에 지도를 그립니다.

<Callout type="info">
이 챕터에는 코드가 한 줄도 나오지 않습니다. 노트와 펜만 있으면 됩니다.
</Callout>

## 왜 이걸 먼저 하나요?

어제 강의를 들으면서 이런 생각, 한 번쯤 드셨을 겁니다.

- "Lovable이 알아서 다 해주는 거 아닌가?"
- "백엔드랑 프론트엔드가 정확히 뭔지 잘 모르겠다."
- "데이터베이스를 왜 따로 둬야 하지? 그냥 화면에 저장하면 안 되나?"

오늘 Lovable Cloud(엔진은 Supabase)를 붙이고, 로그인을 만들고, 배포까지 가려면 **지금 내가 어디를 건드리고 있는지** 감이 잡혀 있어야 합니다. 안 그러면 에러가 났을 때 "어디서 난 에러인지"부터 막힙니다.

## 다시 정리 — 식당으로 비유하면

```
┌─────────────────────────────────────────────────┐
│  프론트엔드     │   백엔드      │   데이터베이스   │
│  (홀)          │   (주방)      │   (냉장고)       │
│                │              │                  │
│  - 메뉴판      │  - 주문 받기  │  - 식재료 보관   │
│  - 손님 응대   │  - 요리       │  - 재고 관리     │
│  - 결제 받기   │  - 음식 내주기 │                  │
└─────────────────────────────────────────────────┘
```

| 영역 | 식당으로 치면 | 우리 앱에서 | 오늘 쓰는 도구 |
| --- | --- | --- | --- |
| **프론트엔드** | 손님이 보는 홀 | 사용자가 보는 화면, 버튼, 입력창 | Lovable로 만든 것 |
| **백엔드** | 주방 | 데이터 처리, 로그인 검사, 권한 확인 | Lovable Cloud가 대신 함 |
| **데이터베이스** | 냉장고 | 데이터가 실제로 쌓이는 창고 | Lovable Cloud 안의 Postgres |

### 한 문장으로

- **프론트엔드** = 사용자가 **보고 만지는 곳**
- **백엔드** = 화면 뒤에서 **판단하고 처리하는 곳**
- **데이터베이스** = 데이터가 **계속 쌓여 있는 곳**

## 한 가지 동작이 어떻게 흘러가는지 — 운동 기록 앱 예시

사용자가 "벤치프레스 60kg 10회"를 입력하고 **저장 버튼**을 눌렀다고 합시다.

```
[사용자]
  ↓ "저장" 클릭
[프론트엔드]
  - 입력창 값 가져오기 (벤치프레스, 60, 10)
  - 빈 칸 없는지 확인
  - 백엔드에 "이거 저장해 줘" 요청 보내기
  ↓
[백엔드]
  - "이 사람 로그인된 사람 맞아?" 확인
  - "이 데이터 형식 맞아?" 검증
  - DB에 "새 행 추가해 줘" 명령
  ↓
[데이터베이스]
  - workouts 테이블에 새 행 1개 추가
  - "추가 완료" 응답
  ↓
[백엔드]
  - 프론트엔드에 "성공" 응답
  ↓
[프론트엔드]
  - "저장되었습니다" 메시지 표시
  - 입력창 비우기
```

이걸 "**한 번의 요청-응답 사이클**"이라고 부릅니다. 우리 앱에서 일어나는 거의 모든 동작이 이 흐름을 따릅니다.

<Callout type="info">
**중요한 포인트**: 프론트엔드는 데이터베이스에 **직접** 말을 걸지 않습니다. 항상 백엔드를 거칩니다. 그래야 "로그인 검사", "권한 확인" 같은 안전 장치가 들어갈 수 있습니다.
</Callout>

## 실습 — 내 앱 3등분 해보기

이제 본인이 어제 만든 앱을 직접 쪼개 봅니다. 노트를 펴고 따라와 주세요.

### 준비물

- 어제 작성한 [내 앱 컨셉 정의서](/day2/concept) (PRD)
- 어제 만든 Lovable 프로젝트 (열어 두기)
- 노트 + 펜 (또는 메모 앱)

### 1단계 — 떠오르는 대로 적기 (10분)

표를 "잘" 채우려고 할 필요 없습니다. **빈 종이에 영역 이름 3개**만 적어 두고, 본인 앱을 쓸 때 어떤 일이 일어나는지 떠오르는 대로 한 줄씩 적어 보세요.

```
프론트엔드 (사용자가 보는 거)
- ...
- ...

백엔드 (화면 뒤에서 처리하는 거)
- ...
- ...

데이터베이스 (데이터가 쌓이는 곳)
- ...
```

각 영역마다 **1~3개**만 적어도 충분합니다. 헷갈리는 건 비워 두고 강사한테 질문하세요.

#### 작성 예시 — 운동 기록 앱

먼저 본인 앱을 한 문단으로 떠올려 봅니다.

> "내 앱은 헬스장에서 운동할 때 무게랑 횟수를 적어두는 앱이야. 로그인해서 내 기록만 보고, 일주일 단위로 얼마나 운동했는지 보고 싶어."

이걸 영역별로 옮겨 적으면 이렇게 됩니다.

**프론트엔드 — 사용자가 보는 거**
- 로그인 화면
- 운동 입력하는 칸 (운동 이름, 무게, 횟수)
- 오늘 기록 목록
- 한 주 통계 그래프

**백엔드 — 화면 뒤에서 처리하는 거**
- "이 사람 로그인된 사람 맞아?" 확인
- 입력받은 운동 기록을 저장 명령
- 본인 기록만 골라서 화면에 돌려주기

**데이터베이스 — 데이터가 쌓이는 곳**
- 회원 정보 (이메일, 가입한 날짜)
- 운동 기록 (운동 이름, 무게, 횟수, 운동한 날짜)

#### 작성 예시 — 일기 앱

> "매일 짧은 일기를 쓰는 앱이야. 내 일기만 보이게 하고 싶고, 기분도 같이 기록하고 싶어."

**프론트엔드**
- 로그인 화면
- 일기 쓰는 칸 (제목, 본문, 오늘 기분)
- 지난 일기 목록

**백엔드**
- 로그인 처리
- "이 일기 글쓴이가 본인 맞아?" 검사
- 일기 저장·수정·삭제 요청 처리

**데이터베이스**
- 회원 정보
- 일기 (제목, 본문, 기분, 작성한 날짜, 누가 썼는지)

<Callout type="info">
**완벽하게 안 적혀도 괜찮습니다.** 처음에는 "이게 프론트인지 백엔드인지 헷갈리는 항목"이 한두 개 있어야 정상입니다. 그게 오늘 우리가 같이 풀 부분이에요.
</Callout>

### 2단계 — 핵심 동작 1개의 흐름 그리기 (10분)

본인 앱에서 **가장 중요한 동작 1개**를 골라, 위에서 본 화살표 그림처럼 흐름을 그려 봅니다.

예를 들어:

- 운동 기록 앱 → "운동 1개 추가하기"
- 일기 앱 → "오늘 일기 저장하기"
- 책 추천 앱 → "내가 좋아하는 책 등록하기"

그림 형식:

```
[사용자] → 어떤 동작?
  ↓
[프론트엔드] → 무엇을 하는가? (3줄 이내)
  ↓
[백엔드] → 무엇을 검사하고 무엇을 시키는가?
  ↓
[데이터베이스] → 어디 테이블에 무엇을 저장/조회하는가?
  ↓ (응답)
[백엔드] → 무엇을 돌려주는가?
  ↓
[프론트엔드] → 사용자에게 무엇을 보여주는가?
```

<Callout type="info">
**막혔을 때 팁**: 식당으로 옮겨 생각해 보세요. "손님이 메뉴 주문 → 종업원이 받음 → 주방에 전달 → 냉장고에서 재료 꺼냄 → 요리 → 손님에게 서빙"
</Callout>

### 3단계 — 자가 점검 (5분)

본인이 그린 표와 흐름을 보면서 다음 질문에 답해 보세요.

- [ ] 내 앱의 **프론트엔드**에는 화면 / 버튼 / 입력창만 들어가 있는가? (데이터 저장 같은 게 끼어 있진 않은가?)
- [ ] 내 앱의 **데이터베이스**에는 "표(테이블)"만 들어가 있는가? (화면이나 로직이 끼어 있진 않은가?)
- [ ] 내 앱의 **백엔드**가 빠져 있을 때, 프론트와 DB가 직접 말을 걸면 어떤 문제가 생길까?
- [ ] 내가 그린 흐름에서 **로그인 검사**는 어디에서 일어나는가?

## 자주 헷갈리는 것들

### Q1. "로그인은 어디서 처리되나요?"

**답: 거의 모두 백엔드에서 처리됩니다.** 프론트엔드는 "이메일·비밀번호 입력칸"만 보여주고, 입력값을 백엔드에 넘깁니다. "이 비밀번호 맞아?" 판단은 백엔드(Lovable Cloud Auth)가 합니다. 비밀번호 검증 로직이 사용자 화면 코드에 노출되지 않으므로, 누구도 우회해서 다른 사람의 계정에 들어갈 수 없습니다.

### Q2. "이미지 업로드는 어디에 저장되나요?"

**답: 데이터베이스가 아니라 별도 저장소(Storage).** 이미지·파일 같은 큰 데이터는 DB에 직접 넣지 않고 **Storage**라는 곳에 따로 저장합니다. DB에는 "이미지 위치" 주소(URL)만 적어 둡니다. Lovable Cloud Storage(엔진은 Supabase Storage)가 그 역할을 합니다.

### Q3. "통계 계산은 어디서 하나요?"

예: "이번 주 총 운동량 보여줘"

**답: 보통 백엔드에서.** DB에 모든 기록이 있으니 백엔드가 DB에 "이번 주 행만 모아서 합계 내 줘"라고 명령합니다. 프론트엔드는 결과만 받아서 그래프로 그립니다.

> 작은 통계라면 프론트엔드에서도 가능하지만, **데이터가 많아지면 무조건 백엔드 처리**가 정답입니다.

### Q4. "Lovable이 만든 화면은 프론트엔드만인가요?"

**답: 거의 그렇습니다.** Lovable은 화면(프론트엔드)을 만들어 줍니다. 데이터를 임시로 화면 안에 들고 있을 수는 있지만, 새로고침하면 사라집니다. 그래서 오늘 백엔드(Lovable Cloud)를 붙여서 **진짜로 저장**되게 만드는 겁니다.

### Q5. "Lovable Cloud는 백엔드인가요, 데이터베이스인가요?"

**답: 둘 다 합쳐 놓은 도구입니다.** Lovable Cloud(엔진은 Supabase) 안에는:

- **인증(Auth)** — 로그인 처리 (백엔드 역할)
- **DB(Postgres)** — 데이터 저장 (DB 역할)
- **Storage** — 파일 저장 (별도 저장소)
- **API** — 프론트가 부를 수 있는 통로 (백엔드 역할)

그래서 우리는 Lovable Cloud를 켜는 것만으로 **백엔드 + DB**를 동시에 얻습니다. 이게 BaaS의 핵심입니다.
