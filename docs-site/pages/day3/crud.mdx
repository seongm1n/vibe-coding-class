# 데이터 저장하고 불러오기 (CRUD 한 사이클)

## 이 챕터의 목표

본인 앱에서 **사용자가 입력한 데이터가 진짜로 저장**되고, 다음에 다시 들어와도 그대로 보이게 만듭니다.

이걸 한번 끝내면 비로소 "진짜 앱"이 됩니다.

## "CRUD"가 뭔가요?

**CRUD** 는 데이터로 할 수 있는 4가지 동작의 첫 글자를 딴 말입니다.

| 약자 | 단어 | 우리말 | 예시 |
| --- | --- | --- | --- |
| **C**reate | 만들기 | 추가하기 | 새 운동 기록 입력 |
| **R**ead | 읽기 | 불러오기 | 이번 주 기록 목록 보기 |
| **U**pdate | 갱신하기 | 수정하기 | 기록 내용 고치기 |
| **D**elete | 삭제하기 | 지우기 | 잘못 입력한 기록 삭제 |

오늘은 4개를 모두 한 사이클 돕니다.

## 작업 흐름

```
1. Supabase에 테이블 만들기 (어떤 데이터를 담을지)
   ↓
2. RLS 정책 추가 ("내 데이터만 보이게")
   ↓
3. Lovable에 CRUD 추가 시키기
   ↓
4. 미리보기에서 입력 → 새로고침 → 그대로 보이는지 확인
```

## 단계 1 — Supabase에 테이블 만들기

Day 2 숙제에서 적어 온 **저장될 데이터**를 테이블로 만듭니다.

### 예시 — 운동 기록 테이블

```
[ workouts 테이블 ]

| 열 이름     | 타입       | 설명                          |
|-------------|------------|-------------------------------|
| id          | uuid       | 자동 생성 (고유 번호)         |
| user_id     | uuid       | 어느 사용자의 기록인지        |
| date        | date       | 운동 날짜                     |
| exercise    | text       | 운동 종목 (예: "벤치프레스")  |
| weight      | int        | 무게 (kg)                     |
| reps        | int        | 횟수                          |
| memo        | text       | 메모 (선택)                   |
| created_at  | timestamp  | 자동 (생성 시각)              |
```

### 테이블 만드는 방법 — Lovable이 알아서 함

좋은 소식: **테이블을 직접 만들 필요가 거의 없습니다.** 다음 프롬프트를 던지면 Lovable이 Supabase에 테이블까지 자동으로 만들어 줍니다.

```text filename="✅ CRUD 추가 프롬프트 (운동 기록 예시)"
로그인한 사용자가 본인 운동 기록을 추가/조회/수정/삭제할 수 있게 해 줘.

데이터 구조 (Supabase에 workouts 테이블로 만들어 줘):
- id (uuid, 자동 생성)
- user_id (uuid, 로그인된 사용자의 ID)
- date (date)
- exercise (text)
- weight (int)
- reps (int)
- memo (text, 선택)
- created_at (timestamp, 자동)

화면 동작:
1) 메인 화면 입력 폼에서 저장 버튼을 누르면 workouts 테이블에 INSERT
2) "이번 주 기록" 화면에서 본인 user_id의 데이터만 불러와 표시 (날짜 내림차순)
3) 각 기록 카드에 "수정", "삭제" 버튼 (수정은 모달로, 삭제는 확인 후)
4) 데이터가 없을 땐 "아직 기록이 없어요" 빈 상태 화면

보안:
- Supabase RLS 정책을 활성화
- 본인 user_id의 데이터만 SELECT/INSERT/UPDATE/DELETE 가능
- 다른 사용자의 데이터는 보이지도, 수정되지도 않게

추가:
- 데이터 저장/조회 중에는 로딩 스피너
- 에러가 발생하면 사용자에게 친절한 메시지로 표시
```

이 프롬프트 한 번이면 Supabase 테이블 + RLS 정책 + 화면 연결까지 모두 됩니다.

## 단계 2 — Supabase에서 테이블 확인

Lovable이 작업을 끝낸 뒤, Supabase 대시보드 → **Table Editor**로 가면 새 테이블이 보입니다.

```
Table Editor
├── workouts  ← 새로 생긴 테이블
│   └── (열들: id, user_id, date, exercise, weight, reps, memo, created_at)
```

테이블 이름을 클릭하면 빈 표가 보입니다. 다음 단계에서 입력하면 여기에 행이 쌓입니다.

## 단계 3 — RLS 정책 확인

테이블 우측 상단 **"Authentication"** 탭(또는 자물쇠 아이콘)을 클릭하면 RLS 정책이 보입니다.

올바른 RLS 정책 예시:

```sql
-- 본인 데이터만 SELECT
CREATE POLICY "own data select"
ON workouts FOR SELECT
USING (auth.uid() = user_id);

-- 본인 데이터만 INSERT
CREATE POLICY "own data insert"
ON workouts FOR INSERT
WITH CHECK (auth.uid() = user_id);

-- 본인 데이터만 UPDATE
CREATE POLICY "own data update"
ON workouts FOR UPDATE
USING (auth.uid() = user_id);

-- 본인 데이터만 DELETE
CREATE POLICY "own data delete"
ON workouts FOR DELETE
USING (auth.uid() = user_id);
```

<Callout type="info">
**SQL을 직접 적을 필요는 없습니다.** Lovable이 위 프롬프트로 자동 생성해 줍니다. 위 코드는 "이런 게 만들어졌구나" 정도로만 보세요.
</Callout>

## 단계 4 — 미리보기에서 한 사이클 시도

미리보기에서 다음을 차례로 시도하세요.

### Create — 추가

1. 메인 화면에서 운동 1개 기록 입력 (예: 벤치프레스 60kg 10회)
2. 저장 버튼 → "저장되었습니다" 메시지

### Read — 불러오기

1. "이번 주 기록" 화면 이동
2. 방금 입력한 기록이 카드로 보이는지 확인

### Update — 수정

1. 카드의 "수정" 버튼 클릭
2. 횟수를 8회로 변경 → 저장
3. 화면에 8회로 반영되는지 확인

### Delete — 삭제

1. 카드의 "삭제" 버튼 클릭
2. "정말 삭제하시겠습니까?" 확인 → 삭제
3. 카드가 사라지고 "기록이 없어요" 빈 상태가 보이면 OK

### 진짜 저장됐는지 마지막 점검

**브라우저 새로고침 (F5)** 후에도 데이터가 그대로 있으면 진짜 DB에 저장된 것입니다. 그렇지 않으면 어딘가 잘못된 거예요.

Supabase Table Editor에서 직접 행을 확인할 수도 있습니다.

## 흔한 에러 + 해결

### 에러 1: 저장은 되는데 다시 들어오니 안 보임

**원인**: 입력 시 `user_id`가 안 들어감 → RLS가 막음

**해결 프롬프트**:
```text
INSERT 시 user_id가 자동으로 현재 로그인된 사용자의 ID(supabase.auth.getUser())로 설정되게 해 줘.
```

### 에러 2: "permission denied" 또는 "row level security policy"

**원인**: RLS 정책이 없거나 잘못됨

**해결**:
1. Supabase Table Editor → workouts 테이블 → Authentication 탭
2. 정책이 4개(SELECT/INSERT/UPDATE/DELETE) 있는지 확인
3. 없으면 다음 프롬프트로 다시 시도
```text
workouts 테이블에 RLS 정책이 빠진 것 같아.
본인 user_id 데이터에만 SELECT/INSERT/UPDATE/DELETE 가능하도록 정책 4개 추가해 줘.
```

### 에러 3: 데이터는 들어가는데 화면이 비어 있음

**원인**: Read 쿼리가 잘못됐거나 RLS가 너무 엄격함

**해결 1**: 새로고침 후 재시도
**해결 2**: Supabase Table Editor에서 직접 데이터가 있는지 확인 → 있다면 화면 코드 문제
```text
"이번 주 기록" 화면에서 데이터가 화면에 보이지 않아.
- workouts 테이블에서 user_id가 현재 로그인된 사용자인 행만 가져오기
- date 내림차순 정렬
- 콘솔에 받아온 데이터를 console.log로 찍어 줘
```

### 에러 4: 다른 사람 데이터가 보임

**원인**: RLS가 꺼져 있거나 정책이 잘못됨

**해결**: 즉시 다음 프롬프트로 수정
```text
workouts 테이블에서 다른 사용자의 데이터가 보이는 보안 문제가 있어.
- RLS를 켜고
- SELECT 정책에 auth.uid() = user_id 조건을 적용
- 빠르게 수정해 줘
```

## 진도 빠른 사람용 — 추가 시도

기본 CRUD가 끝났다면 다음을 시도해 보세요.

### 검색·필터

```text
"이번 주 기록" 화면 상단에 검색창 추가.
- 운동 종목 텍스트로 검색
- 부위(가슴/등/하체...)로 필터
```

### 통계

```text
메인 화면 상단에 다음 통계 카드 추가:
- 이번 주 운동 횟수
- 이번 달 총 무게 합계 (weight × reps)
- 가장 자주 한 운동 1개
```

### 이미지 업로드

```text
운동 기록에 이미지 첨부 기능 추가.
Supabase Storage에 업로드, 썸네일은 카드에 표시.
RLS는 본인이 올린 이미지만 보이게.
```

## 진도 느린 사람용 — 우선순위

CRUD 4개 중 **Create + Read** 만 되어도 OK 입니다. Update·Delete는 나중에 추가할 수 있습니다.

```text filename="간단 CRUD 프롬프트 (느린 사람용)"
일단 Create와 Read만 먼저 만들어 줘:
- 메인 화면 폼에서 입력 → workouts 테이블 저장 (user_id 자동)
- "이번 주 기록" 화면에서 본인 데이터만 불러와 표시
- RLS 정책 (SELECT, INSERT만)

수정/삭제는 다음 단계에서 추가할게.
```

## 체크포인트

- [ ] Supabase Table Editor에 본인 테이블이 보임 (예: workouts)
- [ ] RLS 정책이 4개(또는 최소 SELECT/INSERT 2개) 있음
- [ ] 미리보기에서 데이터 입력 → 저장 → **새로고침 후에도 그대로**
- [ ] Supabase Table Editor에서 행이 쌓이는 게 보임
- [ ] 다른 사용자(테스트 계정)로 로그인하면 본인 데이터가 안 보임

## 자주 묻는 질문

### Q. "내 데이터 다른 사람이 볼 수 있나요?"

RLS를 켜고 정책이 제대로 들어 있다면 못 봅니다. 위 체크포인트의 마지막 항목으로 직접 확인하세요. 다른 이메일로 가입한 두 번째 계정에서 로그인 → 본인 데이터가 안 보이는지 확인.

### Q. "데이터를 엑셀로 받을 수 있나요?"

Supabase Table Editor에서 테이블을 선택 → 우측 상단 **Export** → CSV 다운로드. 엑셀로 열 수 있습니다.

### Q. "용량 제한이 있나요?"

무료 플랜은 **DB 500MB, Storage 1GB** 입니다. 텍스트 데이터는 거의 못 채웁니다. 이미지/파일을 많이 올릴 계획이면 그때 가서 유료 검토하세요.
