# (번외) 직접 Supabase·Vercel 운영하기

<Callout type="info">
이 챕터는 **번외 자료**입니다. 강의 본 흐름은 [Lovable Cloud](/day3/lovable-cloud) + [Lovable 배포](/day3/deploy)로 진행됩니다. 이 페이지는 강의 후 "내 백엔드를 진짜 내 것으로 만들고 싶다"는 분들을 위한 가이드입니다.
</Callout>

## 왜 이걸 따로 보나요?

Lovable Cloud는 빠르고 편합니다. 하지만 다음과 같은 경우에는 **본인이 Supabase·Vercel을 직접 운영**하는 쪽이 나을 수 있습니다.

- **데이터 소유권**: Lovable 계정에 묶이지 않고 어디든 옮길 수 있는 백엔드를 원할 때
- **비용 예측**: 사용자가 많아져 Cloud 한도를 자주 넘길 때 (Supabase 직접 운영이 더 쌀 수 있음)
- **고급 기능**: Supabase의 Edge Functions, 복잡한 SQL 쿼리, 복수 환경(개발/운영) 분리 등을 본격적으로 다룰 때
- **이식성**: GitHub 코드 + 외부 호스팅(Vercel/Netlify/자체 서버) 조합으로 갈 계획일 때

수강 단계에선 Lovable Cloud로 충분합니다. **이 페이지는 "다음 단계"** 로만 참고하세요.

## 전체 흐름

```
1. Supabase에 본인 프로젝트 만들기 (가입 → 프로젝트 생성)
   ↓
2. Lovable이 사용하는 Cloud 연결을 본인 Supabase로 교체
   ↓
3. Lovable에서 GitHub로 코드 익스포트
   ↓
4. Vercel에 GitHub 레포 임포트 + 환경 변수 등록
   ↓
5. Vercel 배포 → 본인 도메인 사용
```

각 단계를 풀면 다음과 같습니다.

## 1. Supabase 프로젝트 만들기

### 가입

1. [https://supabase.com](https://supabase.com) 접속
2. **"Start your project"** → **"Continue with GitHub"** 로 가입

### 새 프로젝트

1. 대시보드 → **"New project"**
2. 다음 정보 입력:

| 항목 | 입력값 |
| --- | --- |
| Project name | 예: `fitcheck`, `my-app` (영어 소문자 + 하이픈) |
| Database Password | 자동 생성 버튼 → **메모해 두기** |
| Region | **Northeast Asia (Seoul)** |
| Pricing Plan | **Free** |

3. **"Create new project"** 클릭 → 1~3분 대기

### API 키 두 개 메모

프로젝트 생성 후 좌측 **⚙️ Settings → API** 메뉴에서:

```text filename="메모할 값"
Project URL = https://xxxxxxx.supabase.co
Publishable Key (구 anon key) = eyJxxxxxxx...

(Secret key / service_role 키는 무시 — 절대 공개 금지)
```

<Callout type="warning">
**최근 Supabase 키 이름 변경**: `anon` → **Publishable key**, `service_role` → **Secret key** 로 표기됩니다. **Publishable key**가 프론트엔드에 안전하게 노출되는 키입니다.
</Callout>

## 2. Lovable의 Cloud 연결을 본인 Supabase로 교체

Lovable 채팅창에 다음을 보냅니다.

```text filename="✅ 외부 Supabase로 교체 프롬프트"
지금 Lovable Cloud의 Supabase가 아니라, 내 개인 Supabase 프로젝트로 연결을 바꿔 줘.

Supabase URL: https://xxxxxxx.supabase.co
Supabase Publishable Key: eyJxxxxxxx...

작업:
- src/integrations/supabase/client.ts의 환경 변수가 위 값을 사용하도록 .env 갱신
- 기존 Lovable Cloud 연결은 제거
- 회원가입/로그인/CRUD 코드는 그대로 유지 (Supabase API는 호환되니까)

확인용으로 client.ts와 .env를 수정 후 보여 줘.
```

이 작업이 끝나면 본인 Supabase 대시보드 → Authentication > Users에서 가입자가 직접 보입니다.

<Callout type="error">
**기존 데이터는 함께 옮겨지지 않습니다.** Lovable Cloud에서 만든 사용자·기록은 Lovable Cloud에 남고, 새 본인 Supabase는 빈 상태로 시작합니다. 데이터 이전이 필요하다면 별도 export/import 작업이 필요해요.
</Callout>

## 3. Lovable에서 GitHub로 코드 익스포트

Lovable에는 **"Connect to GitHub"** 또는 **GitHub 익스포트** 기능이 있습니다.

1. Lovable 우측 상단 → **GitHub 아이콘** 또는 **"Export to GitHub"**
2. GitHub 권한 승인
3. 레포지토리 이름 입력 (예: `fitcheck-app`)
4. Lovable이 본인 GitHub에 새 레포지토리 + 코드를 푸시

본인 GitHub에 가면 새 레포가 보입니다.

<Callout type="info">
**Public/Private**: 환경 변수(키)는 코드에 안 들어 있으니 Public이어도 안전합니다. 다만 **본인 코드를 남이 무단 복제할 수 있다는 점**은 인지하고 결정.
</Callout>

## 4. Vercel에 GitHub 레포 임포트

### Vercel 가입

1. [https://vercel.com](https://vercel.com) 접속 → **"Continue with GitHub"** 로 가입

### 프로젝트 임포트

1. Vercel 대시보드 → **"Add New..."** → **"Project"**
2. **Import Git Repository** 섹션에 본인 레포가 보임
3. **"Import"** 클릭

레포가 안 보이면 **"Adjust GitHub App Permissions"** → Vercel 접근 가능 레포에 본인 레포 추가.

### 환경 변수 등록 (가장 중요)

Import 화면 중간에 **Environment Variables** 섹션이 있습니다.

| Name | Value |
| --- | --- |
| `VITE_SUPABASE_URL` | (본인 Supabase Project URL) |
| `VITE_SUPABASE_PUBLISHABLE_KEY` | (본인 Publishable Key) |

<Callout type="warning">
**환경 변수 이름**: Lovable이 코드에서 사용한 이름과 정확히 같아야 합니다. `src/integrations/supabase/client.ts` 또는 `.env.example`을 보고 정확한 이름 확인. Vite 기반이면 `VITE_`, Next.js 기반이면 `NEXT_PUBLIC_` 접두사가 붙어 있을 거예요.
</Callout>

### Deploy 클릭

환경 변수까지 입력했으면 **"Deploy"** 버튼 클릭 → 1~3분 대기 → `xxx.vercel.app` URL 발급.

## 5. 배포 후 추가 작업

### Site URL 등록

본인 Supabase 대시보드 → **Authentication → URL Configuration**:

- **Site URL**: `https://xxx.vercel.app` (Vercel 배포 URL)
- **Redirect URLs**: 위 URL을 그대로 또는 `https://xxx.vercel.app/**` 로 추가

이걸 안 하면 회원가입·비밀번호 재설정 이메일의 링크가 잘못된 곳으로 갑니다.

### 자동 재배포

GitHub에 새 커밋이 푸시될 때마다 Vercel이 자동으로 새 버전을 배포합니다. Lovable에서 화면 수정 → GitHub 자동 푸시 → Vercel 자동 재배포 흐름이 작동.

## 흔한 함정

### 1. 환경 변수 이름 불일치

Lovable이 사용한 이름(예: `VITE_SUPABASE_URL`)과 Vercel에 등록한 이름이 다르면 배포본에서 Supabase 연결이 안 됩니다. **글자 하나라도 다르면 안 됨.**

### 2. URL 끝의 슬래시

```
잘못: https://xxxxx.supabase.co/
맞음: https://xxxxx.supabase.co
```

### 3. RLS 정책 누락

Lovable Cloud에서 자동으로 만들어졌던 RLS 정책이 **본인 Supabase 프로젝트에는 없을 수 있습니다.** 채팅에 한 번 더 시켜 주세요:

```text
내 개인 Supabase에 workouts 테이블의 RLS 정책 4개(SELECT/INSERT/UPDATE/DELETE)
다시 만들어 줘. 본인 user_id 데이터만 접근 가능한 정책으로.
```

### 4. CORS 에러

Supabase가 새 Vercel 도메인을 인식 못 하면 빨간 CORS 에러가 뜹니다. 위의 "Site URL 등록" 단계로 해결.

## 비교 — Lovable Cloud vs 직접 운영

| 항목 | Lovable Cloud | 직접 (Supabase + Vercel) |
| --- | --- | --- |
| 셋업 시간 | 5분 | 30분~1시간 |
| 무료 한도 | 워크스페이스당 월 $25 | Supabase Free + Vercel Hobby (각 별도) |
| 데이터 소유 | Lovable 계정 종속 | 본인 Supabase 계정 |
| 도메인 | `xxx.lovable.app` 또는 커스텀 | `xxx.vercel.app` 또는 커스텀 |
| 자동 재배포 | 수동 Publish 필요 | GitHub 푸시 → 자동 |
| 학습 곡선 | 낮음 | 중간 (환경 변수·DNS 등) |
| 다른 호스팅으로 이전 | 어려움 | 쉬움 |

## 언제 옮기는 게 합리적?

다음 신호가 보이면 직접 운영을 검토하세요.

- [ ] 매월 사용자가 늘어 Lovable Cloud 한도를 정기적으로 초과
- [ ] 회사·팀 단위로 운영하려는데 Lovable 계정이 1인 소유라 곤란
- [ ] Edge Functions, Cron, 외부 API 통합 등 고급 기능을 본격적으로 사용
- [ ] 데이터 백업·이전 정책을 본인이 직접 통제해야 함

수강 직후 / 친구 1~2명에게 보여 주는 단계라면 **Lovable Cloud 그대로 쓰는 게 합리적**입니다.

## 정리

- 강의 본 흐름은 **Lovable Cloud + Lovable Publish** 로 충분
- 운영 단계에서 더 큰 통제가 필요하면 본인 Supabase + Vercel로 마이그레이션 가능
- 마이그레이션 시 가장 큰 함정은 **환경 변수 이름 일치 + RLS 정책 재생성**

추가 도구 정보는 [도구·참고자료 모음](/resources)을, 다른 BaaS 비교는 [(번외) Supabase vs Firebase](/extra-firebase)를 참고하세요.
