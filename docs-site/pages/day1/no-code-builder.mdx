# 노코드 앱 빌더 미리보기

## 이 챕터의 목표

내일(Day 2)부터 본격적으로 사용할 **노코드 앱 빌더 Lovable**을 다 같이 한 번 만져 봅니다. 한 줄짜리 프롬프트로 화면이 나오는 걸 본인 노트북에서 직접 경험합니다. 비교용으로 v0와 Bolt도 짧게 살펴봅니다.

오늘은 **가볍게 한 화면**만 띄워 보면 됩니다. 본격적인 본인 앱 만들기는 내일입니다.

## "노코드 앱 빌더"가 뭔가요?

**노코드(No-Code)** 는 말 그대로 "코드가 없는" 도구입니다. 마우스 클릭과 자연어 지시만으로 앱을 만들 수 있게 해 줍니다.

비유하자면, 옛날엔 집을 지으려면 벽돌을 직접 쌓아야 했지만, 이제는 "이런 모양의 집 지어 줘"라고 말하면 자동으로 지어 주는 도구가 생긴 셈입니다.

## 비교: 3개 도구

| 도구 | 만든 곳 | 특징 | 무료 한도 | 잘 맞는 사용자 |
| --- | --- | --- | --- | --- |
| **[Lovable](https://lovable.dev)** | Lovable | 한국어 잘 먹음, 비전공자 친화적 | 일/월 메시지 한도 | **이 강의의 메인 도구** |
| [v0](https://v0.dev) | Vercel | UI 디자인에 강함, 코드 품질 우수 | 일정 메시지 무료 | UI 디자인이 중요한 앱 |
| [Bolt](https://bolt.new) | StackBlitz | 풀스택 코드 생성, 백엔드도 자동 | 일정 토큰 무료 | 더 깊이 들어가고 싶을 때 |

<Callout type="info">
**왜 Lovable을 메인으로 쓰나요?**

- 한국어로 시켜도 결과가 잘 나옵니다.
- UI가 비전공자에게 가장 직관적입니다.
- Supabase(백엔드) 연동이 한 번 클릭으로 됩니다 → Day 3에서 활용합니다.
- Vercel/GitHub 배포가 쉽습니다 → Day 3 마지막에 배포합니다.
</Callout>

## Lovable이 처리하는 4단계 흐름

다 같이 따라 하기 전에, Lovable이 무엇을 어떻게 처리하는지 흐름만 한 번 짚고 갑니다. 이 흐름은 잠시 후 본인 노트북에서 직접 경험합니다.

### 1단계 — 프롬프트로 첫 화면 만들기

Lovable 새 프로젝트에 자연어로 시키면 1~2분 안에 화면이 나옵니다. 디자인·구성·간단한 동작이 모두 자동으로 들어갑니다.

### 2단계 — 수정 요청하기

처음 결과는 100% 만족스럽지 않습니다. 채팅창에 추가로 시킵니다. 핵심은 **한 메시지에 여러 수정을 묶어서** 시키는 것입니다. 무료 메시지 한도를 아낄 수 있습니다.

### 3단계 — 결과 확인 + 되돌리기

오른쪽 패널에서 **실시간 미리보기**가 보입니다. 마음에 안 들면 "이전 버전으로 되돌리기"가 가능합니다 (마치 워드의 "실행 취소"처럼).

### 4단계 — 배포 (Day 3에 함)

준비가 되면 한 번 클릭으로 인터넷에 띄울 수 있습니다. 오늘은 여기까지는 안 합니다.

## v0과 Bolt 짧은 비교

### v0 (Vercel)

```text
강점: UI 디자인 결과가 가장 깔끔. 컴포넌트 단위로 만들기 좋음.
약점: 한국어가 영어보다 결과가 살짝 떨어짐. 풀앱보다 화면 단위에 강함.
무료: 일정량 메시지 무료.
```

### Bolt (StackBlitz)

```text
강점: 더 복잡한 앱·백엔드 코드까지 같이 만들어 줌.
약점: 비전공자에겐 정보가 너무 많아 보일 수 있음.
무료: 일/월 토큰 단위 무료.
```

## "왜 Lovable이 막혔을 때 백업이 필요해요?"

20명이 동시에 같은 도구를 쓰면 다음 일이 일어날 수 있습니다.

1. **무료 메시지 한도 소진** — 하루 5메시지 정도만 무료입니다.
2. **서버 응답이 느려짐** — WiFi 부하 + 동시 트래픽
3. **결과가 마음에 안 들 때** — 다른 도구가 같은 프롬프트를 더 잘 만들기도 함

이때 **v0** 또는 **Bolt**로 갈아타면 시간을 아낄 수 있습니다.

<Callout type="warning">
**중요**: 도구를 갈아타도 **PRD(요구사항 문서)는 그대로** 사용합니다. PRD를 잘 만들어 두면 어떤 도구로든 비슷한 결과를 낼 수 있습니다. Day 2에 PRD부터 만드는 이유가 이것입니다.
</Callout>

## 다 같이 따라 하기

이 챕터는 **강사가 화면 공유로 한 줄씩 안내하면, 모두가 본인 노트북에서 동시에 따라 하는** 흐름입니다. 강사만 보여주고 끝나는 게 아니라, **모두가 같은 속도로 같은 화면**을 만들어 봅니다.

### 0단계 — Lovable 가입 확인

지금 바로 [https://lovable.dev](https://lovable.dev) 에 접속해서 로그인 상태인지 확인하세요.

- 로그인 안 됐으면 → **"Sign up"** → GitHub 또는 Google
- 로그인 됐으면 → 메인 대시보드가 보입니다

가입에서 막히면 강사·보조 인력에게 즉시 알려 주세요.

### 1단계 — Lovable UI 짧은 투어

강사가 본인 화면에서 **3가지만** 짚습니다. 본인 화면에서도 같은 위치를 찾으세요.

| 짚을 곳 | 본인 화면에서 찾기 |
| --- | --- |
| ① 채팅창 | 보통 좌측 또는 하단. 프롬프트를 입력하는 곳 |
| ② 미리보기 영역 | 보통 우측. 만든 화면이 실시간으로 보이는 곳 |
| ③ 메시지 카운터 | 보통 화면 어딘가에 "오늘 X개 남음" 표시 |

3곳을 본인 화면에서 다 찾았으면 **손 들기**로 신호.

<Callout type="info">
Lovable UI는 가끔 바뀝니다. 위치가 살짝 다르면 강사·보조 인력과 함께 찾으세요. **위치만 알면 나머지는 다 됩니다.**
</Callout>

### 2단계 — 다 같이 첫 프로젝트 만들기

지금부터 강사가 한 줄씩 신호를 줍니다. 본인 노트북에서 동시에 따라 하세요.

#### 진행 흐름 (강사 신호 기준)

```
[ 다 같이 ① ] Lovable 메인의 "+ New Project" 또는 가운데 큰 입력창 클릭
   ↓ 손 들기
[ 다 같이 ② ] 아래 프롬프트를 복사 → 입력창에 붙여넣기
   ↓
[ 다 같이 ③ ] Enter 누르기
   ↓ 1~3분 대기
[ 다 같이 ④ ] 결과 화면이 뜨면 손 들기
   ↓ 모두 손 들 때까지 잠깐 기다림
```

복붙용 프롬프트:

```text filename="Day 1 체험용 — 다 같이 이거 그대로 사용"
오늘의 점심 식당을 추천해 주는 모바일 웹앱을 만들어 줘.

타겟: 세종시에 사는 30대 직장인
화면 구성:
- 화면 상단: "오늘의 점심" 큰 글씨 헤드라인
- 화면 중앙: 추천 식당 카드 3개 (식당 이름, 거리, 메뉴 종류, 가격대)
- 카드 우측 상단: ❤️ 즐겨찾기 아이콘
- 화면 하단 고정: 큰 "다른 추천 보기" 버튼

스타일:
- 미니멀, 둥근 모서리 (12px)
- 색상: 흰 배경, 짙은 주황(#c8401a) 강조
- 한국어, 모바일 우선 (가로 360px)

식당 정보는 가짜 샘플 3개로 채워줘 (한식 / 일식 / 양식 1개씩).
```

<Callout type="info">
**모두 같은 프롬프트를 던지지만, 결과 화면은 사람마다 살짝 다릅니다.** Lovable이 매번 같은 결과를 주지 않기 때문입니다. 본인 결과가 예시와 다르더라도 정상입니다.
</Callout>

#### 결과가 뜨면 다 같이 클릭해 보기

강사 신호에 맞춰 **다 같이 같은 동작**을 합니다.

```
[ 다 같이 ⑤ ] 카드 ❤️ 아이콘 클릭 → 색이 바뀌는지 확인
[ 다 같이 ⑥ ] 미리보기 상단에서 모바일/데스크톱 토글 한 번씩
[ 다 같이 ⑦ ] "다른 추천 보기" 버튼 클릭 → 뭐가 일어나는지 보기
```

같은 프롬프트로 만든 화면인데 사람마다 살짝 다른 동작이 들어가 있을 수 있습니다.

### 3단계 — 다 같이 한 메시지로 크게 바꾸기

이번엔 한 메시지에 **여러 변경을 묶는** 연습입니다. Day 2부터는 메시지 한도 관리가 중요해집니다.

```
[ 다 같이 ① ] 채팅창에 다음 프롬프트 붙여넣기
[ 다 같이 ② ] Enter
   ↓ 1~3분 대기
[ 다 같이 ③ ] 결과 비교 — 색·모서리·간격이 다 한 번에 바뀌었나?
```

```text filename="한 메시지로 4가지 변경 — 다 같이"
다음 4가지를 한 번에 변경해 줘:
- 색상 강조를 짙은 주황 대신 진한 청록(#0f8b8d)으로
- 카드 모서리를 더 둥글게 (20px)
- 카드에 식당 사진 자리 추가 (회색 placeholder 박스)
- 카드 사이 간격 살짝 늘리기 (20px)
```

결과 확인 포인트:

- 색이 청록으로 바뀌었는가?
- 카드 모서리가 더 둥글어졌는가?
- placeholder 박스가 추가됐는가?
- 카드 간격이 더 넓어졌는가?

**4개 중 1~2개만 반영됐다면** Lovable이 일부를 놓친 것입니다. 실제 강의 중에는 흔합니다. 다음 단계에서 되돌리고 다시 시도하는 법을 배웁니다.

### 4단계 — 다 같이 "이전 버전으로 되돌리기"

방금 변경이 마음에 안 들거나 일부만 반영됐다면, **이전 버전으로 한 번 되돌려 봅니다.**

```
[ 다 같이 ① ] 채팅 메시지 위쪽의 ↺ 또는 "Restore" 버튼 찾기
[ 다 같이 ② ] 클릭 → 이전 화면으로 돌아가는지 확인
[ 다 같이 ③ ] 다시 한 번 클릭해서 청록 버전으로 복원도 가능
```

이 기능은 Day 2부터 **메시지를 잘못 썼을 때 되살리는 안전장치** 가 됩니다. 위치 한 번 손에 익혀 두세요.

<Callout type="warning">
**되돌리기는 메시지 한도를 차감하지 않습니다.** 이상한 결과가 나왔을 때 새 메시지로 "원래대로 해 줘"라고 시키지 마세요. 한도만 까먹습니다. **되돌리기 버튼**이 정답입니다.
</Callout>

### 5단계 — v0와 짧게 비교 (선택)

같은 프롬프트를 [v0.dev](https://v0.dev) 에 한 번만 던져 봅니다.

```
[ 다 같이 ① ] 새 탭에서 https://v0.dev 접속, 가입 (Vercel 계정)
[ 다 같이 ② ] 같은 프롬프트(첫 번째 것) 붙여넣기 → Enter
   ↓ 1~2분 대기
[ 다 같이 ③ ] 결과를 Lovable 결과와 나란히 비교
```

비교 포인트:

| 비교 | Lovable | v0 |
| --- | --- | --- |
| UI 깔끔함 | | |
| 한국어 처리 | | |
| 처음 결과의 완성도 | | |
| 메시지 1개로 바뀌는 양 | | |

<Callout type="warning">
**메시지 한도 주의**: v0에서도 1개만 던지고 끝입니다. 비교만 하고 넘어가세요. Day 2부터는 Lovable 메시지가 매우 귀합니다.
</Callout>

## 자주 하는 실수

<Callout type="warning">
**실수 1: 한 메시지에 너무 적게 시킴**
"색을 바꿔 줘" → 결과 → "이번엔 글자 크기를" → 결과 ... 이렇게 가면 무료 한도가 5분 만에 소진됩니다. **한 번에 큰 변화를 묶어서** 시키는 게 효율적입니다.
</Callout>

<Callout type="warning">
**실수 2: 처음부터 너무 크게 시작**
"인스타그램 같은 SNS 만들어 줘"는 너무 큽니다. 첫 메시지는 **하나의 화면**에 집중하세요. 화면이 나온 뒤 다음 화면을 추가합니다.
</Callout>

<Callout type="warning">
**실수 3: 결과가 마음에 안 든다고 새 프로젝트를 또 만듦**
새 프로젝트를 만들면 메시지 한도가 한 번 더 깎입니다. **같은 프로젝트 안에서 수정**하는 게 거의 항상 더 빠릅니다.
</Callout>

## 자주 묻는 질문

### Q. "코드를 봐야 하나요?"

오늘은 No. Day 3부터 살짝 봅니다 (보기만 함). Lovable은 "코드 보기" 토글이 있어서 원하면 볼 수 있고, 안 보고 싶으면 안 봐도 됩니다.

### Q. "이대로 앱스토어에 올릴 수 있나요?"

웹앱이라 앱스토어에는 못 올립니다. 인터넷 주소(URL)로 공유하는 형태입니다. 모바일 앱(iOS·Android)으로 빼는 건 별도 과정이고 강의 범위 밖입니다.

### Q. "Lovable 결과가 마음에 안 들면 어떻게 해요?"

대부분 **PRD가 모호한 게 원인**입니다. Day 2에서 PRD 잘 쓰는 법을 다룹니다. PRD가 잘 나오면 화면도 잘 나옵니다.
